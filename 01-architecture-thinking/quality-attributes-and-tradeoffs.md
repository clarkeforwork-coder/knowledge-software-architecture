# 品質屬性與取捨：沒有最好的架構，只有取捨

## 前言

「這個系統要快、要穩、要安全、還要好維護。」——需求會議上這句話一出，所有人點頭，然後什麼也沒有被決定。因為**全都要，等於什麼都沒說**：這四個目標在架構上會互相打架，而沒有人說出「打架時誰讓誰」。

上一篇〈[什麼是軟體架構](what-is-software-architecture.md)〉的結論是：架構師決定的三件事之一，是品質屬性的優先序。這一篇把它展開——品質屬性是什麼、為什麼它們比功能需求更能決定架構長相、以及怎麼把「要快要穩」這種形容詞，變成可以設計、可以驗收、可以吵架吵出結果的東西。

## 技術背景

### 先破除誤解：品質屬性不是「盡量做好」的形容詞

功能需求（能下單、能退貨）決定系統**做什麼**；品質屬性（quality attributes，又稱非功能需求、架構特性）決定系統**在什麼條件下做到什麼程度**。《Software Architecture in Practice》（Bass, Clements & Kazman）的核心論點正是：**形塑架構的主要是品質屬性，不是功能**——同一套功能，「p99 延遲 200ms」和「p99 延遲 2 秒可接受」會蓋出兩套完全不同的系統；「當機一小時沒關係」和「單節點故障 30 秒內自動恢復」也是。

把品質屬性寫成「盡量快」「高可用」，等於把架構決策外包給每個工程師的想像力。

### 常見品質屬性與它們的架構手段

| 品質屬性 | 問的問題 | 典型架構手段 |
|---|---|---|
| 效能（performance） | 一次請求多快回來 | 快取、索引、非同步化、預運算 |
| 可擴展性（scalability） | 量變十倍還撐得住嗎 | 無狀態化、水平擴展、分片 |
| 可用性（availability） | 壞掉時使用者感覺得到嗎 | 冗餘、故障轉移、降級、隔離 |
| 可修改性（modifiability） | 一個需求要改幾個地方 | 模組化、清晰邊界、鬆耦合 |
| 安全性（security） | 資料會不會被看到／改到 | 認證授權、加密、稽核軌跡 |
| 可測試性（testability） | 能不能便宜地驗證行為 | 依賴注入、介面隔離、確定性設計 |
| 可觀測性（observability） | 出事時多快知道發生什麼 | 結構化日誌、指標、分散式追蹤 |

（ISO/IEC 25010 提供了更完整的正式分類，可當檢查清單用。）

### 品質屬性會互相打架

取捨不是抽象哲學，是具體的工程衝突：

| 這一邊 | 那一邊 | 衝突點 |
|---|---|---|
| 效能（加快取） | 可修改性、正確性 | 快取失效邏輯是新的複雜度與新的 bug 面 |
| 可用性（最終一致） | 一致性 | 分區發生時兩者不可兼得——見規劃中的〈CAP 與 PACELC〉 |
| 安全性（多層驗證） | 效能、易用性 | 每道閘門都是延遲與摩擦 |
| 可擴展性（拆服務） | 簡單性、延遲 | 行程內呼叫變網路呼叫，失敗模式倍增 |

所以「沒有最好的架構」不是謙虛話：**架構就是選擇你願意輸在哪裡**。沒有顯性地選，系統仍然會輸——只是輸在沒人挑過的地方。

### 把形容詞變成場景

SEI 的品質屬性場景（quality attribute scenario）方法，把一句形容詞展開成六個要素：刺激來源、刺激、產物、環境、回應、**回應度量**。日常使用不必背齊六項，抓住精神即可——**每個品質屬性都要有條件與數字**：

- ❌「系統要快」→ ✅「促銷尖峰每秒 500 筆下單時，下單 API p99 < 800ms」
- ❌「系統要高可用」→ ✅「單一節點故障時，30 秒內自動恢復服務，期間錯誤率 < 1%」
- ❌「系統要好維護」→ ✅「新增一種付款方式，只修改付款模組，不動訂單核心」

寫成這樣之後，三件事同時發生：手段可以被推導（每秒 500 筆自然導向壓測與容量規劃）、驗收可以被執行（數字達標與否沒有模糊空間）、爭論可以被仲裁（兩個方案用同一組場景比）。

## 實際案例

需求只有一句話：「營運儀表板的數據要**即時**。」三個工程師會給你三套架構：

- 工程師 A 理解成「秒級」：上 streaming pipeline，事件進來即時聚合——多一整套基礎設施要維運。
- 工程師 B 理解成「別太舊就好」：每五分鐘跑一次預聚合批次——簡單，但「即時」兩個字沒人敢簽字。
- 工程師 C 直接讓儀表板查詢打主庫：最即時，然後月底報表一開，主庫連線被打滿，**下單開始逾時**——為了報表的「即時」，輸掉了交易系統的可用性，而且沒人記得做過這個交換。

正確的第一步不是選方案，是**追問到數字**：「即時是多即時？數據晚五分鐘，會做錯什麼決策嗎？」實務上營運方的答案常常是「……五分鐘沒差，我們一天看三次」。於是品質屬性場景變成：

> 儀表板數據延遲 ≤ 5 分鐘；儀表板查詢不得影響交易庫（交易 API p99 不因報表查詢劣化）。

第二句和第一句同樣重要——它把「不能輸掉什麼」也寫了進去。此時方案 B 加上獨立的查詢副本就是顯然解，streaming pipeline 的複雜度與打主庫的風險都被數字排除了。**一句量化，省掉一套基礎設施，或一次生產事故。**

## 技術優缺點

用顯性的、量化的品質屬性驅動架構：

**優勢**

- **爭論可仲裁**：兩個方案不再比信仰，比的是同一組場景下的表現與成本。
- **驗收有依據**：上線前壓測打的就是場景裡的數字，不是拍腦袋的流量。
- **取捨留下痕跡**：「我們選擇讓報表延遲五分鐘以保護交易庫」是一條可以被未來挑戰的決策，而不是一個沒人知道的預設。

**代價**

- **前期溝通成本**：把「要快」追問成數字，需要跟業務方來回，而對方常常答不出來（此時給出預設值請對方否決，比等待答案有效）。
- **數字會僵化**：場景是當時業務的快照；把 p99 < 800ms 當永恆 KPI 而不隨業務重估，會反過來綁死架構演進。
- **量化不了的仍然存在**：像「概念完整性」「開發體驗」這類屬性難以給數字，不能因為量不了就當作不存在。

**這改變了什麼實務決策**：架構評審的第一個問題從「你用了什麼技術」變成「**你的品質屬性場景是什麼、這個設計輸在哪裡**」。答不出後半句的設計，不是沒有取捨，是取捨還沒被看見。

## 小結

- 品質屬性決定系統「在什麼條件下做到什麼程度」；形塑架構的主要是它，不是功能需求。
- 品質屬性天生互相衝突；架構就是選擇你願意輸在哪裡，沒有顯性選擇的系統會輸在沒人挑的地方。
- 把形容詞變成「條件＋數字」的場景，設計才可推導、驗收才可執行、爭論才可仲裁。
- 場景要同時寫下「要贏什麼」與「不能輸什麼」（報表即時 vs 交易庫不受影響）。
- 品質屬性排好了優先序，系統「應該」長成什麼樣就有了答案——但系統**實際上**會長成什麼樣，還有一股更隱蔽的力量在作用：你的組織結構。下一篇：〈[康威定律：組織結構如何形塑系統](conways-law.md)〉。

## 常見面試題

1. **功能需求和非功能需求，哪個對架構影響大？為什麼？**（提示：同功能不同品質屬性 → 完全不同的架構；舉一組對比就到位）
2. **業務方說「系統要高可用」，你會怎麼接？**（提示：追問到條件與數字——可以停多久、什麼時段、使用者感知到什麼；再談成本曲線）
3. **舉一對互相衝突的品質屬性，以及你實際做過的取捨。**（提示：快取 vs 一致性、拆服務 vs 延遲都是好素材，關鍵是說出「我們選擇輸掉什麼、為什麼可以接受」）
