# 康威定律：組織結構如何形塑系統

## 前言

「為什麼我們有三個會員資料服務？」新人翻著系統清單問。老鳥的回答通常不是技術理由：「因為當年有三個事業部，各做各的。」——組織圖上的線，最後都變成了系統架構圖上的線。

前兩篇談了架構是什麼、品質屬性怎麼排優先序——那是系統「應該」長成的樣子。這一篇談一股更隱蔽也更強大的力量：不管你畫了什麼架構圖，系統**實際上**會長成你的組織溝通結構的形狀。這不是玩笑話，它有名字、有原始論文、有五十年的驗證：康威定律。

## 技術背景

### 先破除誤解：這不是段子，是一篇 1968 年的論文

康威定律常被當成工程師的自嘲梗，但它的出處是 Melvin Conway 於 1968 年發表在 *Datamation* 的論文〈How Do Committees Invent?〉，原文陳述是：

> 任何設計系統的組織，其產出的設計必然是該組織**溝通結構的複本**。
> （"Any organization that designs a system (defined broadly) will produce a design whose structure is a copy of the organization's communication structure."）

「康威定律」這個名字則是 Fred Brooks 在《The Mythical Man-Month》（1975）中引用時所起。它不是觀察到的巧合，而是有機制的必然。

### 為什麼必然：介面長在溝通斷層上

兩個模組要有清晰的介面，前提是負責它們的人**談得攏**。溝通的成本結構決定了介面出現的位置：

- 團隊**內部**溝通便宜（同一個站會、同一個聊天群），所以團隊內的模組邊界模糊也能運作——反正隨時能對齊。
- 團隊**之間**溝通昂貴（要排會議、要開票、要等排程），所以跨團隊的地方自然長出正式介面、契約、版本管理——因為沒有契約就無法協作。

結果：**系統的介面沿著組織的溝通斷層線生長**。你可以在架構圖上畫任何邊界，但只有與組織斷層重合的那些會真的存在；其他的會被日常的便宜溝通慢慢磨掉。

### 順著用：反向康威操作

既然組織結構終將複寫到系統上，就把因果反過來用——先決定想要的系統架構，**再調整組織去配合它**。這就是「反向康威操作」（Inverse Conway Maneuver），由 Thoughtworks 推廣、《Team Topologies》（Skelton & Pais, 2019）系統化：

- 想要清晰的服務邊界？讓每個服務有**唯一的擁有團隊**，團隊對齊業務能力（stream-aligned team），而不是對齊技術層。
- 想要模組間低耦合？先確保團隊之間不需要天天開會才能出貨——**團隊的認知負荷與依賴關係，就是未來系統的耦合度**。

服務邊界怎麼對齊業務能力，正是 DDD 戰略設計要回答的問題——見〈[戰略設計：Bounded Context 與 Context Map](../03-ddd/strategic-design-bounded-context.md)〉。

### 反過來也會發生：架構反噬組織

這條定律是雙向的。接手一套三層巨石系統的新創團隊，很快會發現自己「不得不」分出前端組、後端組、DBA——系統的形狀反過來塑造了組織。遺留系統之所以難改，一半的慣性其實在組織圖上。

## 實際案例

### 案例一：分層組織長出分層巨石

一個常見的組織：前端團隊、後端團隊、DBA 團隊，各有主管、各排各的需求。三年後的系統長相完全可以預測：

```
組織圖                        系統圖
┌──────────┐                ┌──────────────────┐
│ 前端團隊  │  ──────────>   │   前端 SPA 巨石    │
├──────────┤                ├──────────────────┤
│ 後端團隊  │  ──────────>   │   後端 API 巨石    │
├──────────┤                ├──────────────────┤
│ DBA 團隊  │  ──────────>   │  一顆共享大資料庫   │
└──────────┘                └──────────────────┘
```

每個「加一個欄位」的需求都要跨三個團隊、開三張票、等三個排程；而每一層**內部**因為溝通便宜，模組邊界全數糊掉——前端什麼都能 import、後端 service 互相亂叫、資料庫兩百張表誰都能 join 誰。介面只長在團隊之間，一條不多、一條不少，精準複寫組織圖。

### 案例二：把定律當工具的公司

反向康威最著名的實踐者是 Amazon。Werner Vogels 在 2006 年 *ACM Queue* 訪談中描述：服務由小團隊（two-pizza team）完整擁有，開發、部署、維運一手包——「**You build it, you run it.**」組織先被切成高自治的小單位，服務化的架構是隨之長出來的結果；而不是先畫了微服務架構圖，再祈禱組織配合。

對照組同樣常見：公司高喊微服務轉型，拆了八十個服務，但組織還是功能式分工＋集中發版委員會——結果是「分散式巨石」：服務在物理上分開，卻必須一起改、一起測、一起上線。架構圖變了，溝通結構沒變，於是系統的**真實**耦合結構也沒變。拆的是部署單位，不是邊界。

## 技術優缺點

把康威定律納入架構工作（而不是當它不存在）：

**優勢**

- **解釋力**：很多「技術上說不通」的系統怪象（三個會員服務、繞不開的共享庫）在組織圖上一目了然，能對症下藥而不是空改架構。
- **多一個槓桿**：架構師的工具箱從「畫圖、定規範」多了「調整團隊邊界與溝通路徑」，而且後者往往更有效。
- **預測力**：重組前就能預判系統走向——把兩個團隊合併，他們的服務邊界大概率會開始糊掉。

**代價**

- **組織調整又貴又慢**：人不是資源方塊，重組有士氣與信任成本；反向康威不是可以季季操作的手段。
- **架構師的權限邊界**：多數架構師動不了組織圖，只能影響；把一切歸咎於組織也可能成為不改技術債的藉口。
- **不是決定論**：紀律良好的團隊可以在一段時間內抵抗定律（維持與組織不重合的邊界），只是要付出持續的守邊成本。

**這改變了什麼實務決策**：做架構決策時多問一個問題——「**這個邊界，有沒有一個溝通結構在支撐它？**」想拆的服務沒有獨立的擁有團隊，拆完就是分散式巨石；反之，重組團隊時也要意識到：你正在重畫未來的系統架構圖。

## 小結

- 康威定律（Conway, 1968）：系統設計必然複寫組織的溝通結構——機制在於介面只長在溝通昂貴的斷層上。
- 定律是雙向的：組織塑造系統，系統也會反過來固化組織。
- 反向康威操作：先定目標架構，再調整團隊邊界去支撐它；團隊拓撲即未來的系統拓撲。
- 只拆服務不拆組織，得到的是分散式巨石——部署單位變了，耦合結構沒變。
- 架構評審的隱藏題：每一條你想要的邊界，背後都需要一個養得起它的溝通結構。
- 組織決定了邊界能不能活下來，但邊界**本身**該怎麼畫——層怎麼分、依賴朝哪個方向——是架構風格的課題。第一章到此收束，下一章從〈[分層架構與它的邊界](../02-architecture-styles/layered-architecture-and-boundaries.md)〉開始；物件層級的邊界規則已有〈[分層物件家族](../02-architecture-styles/layered-object-family.md)〉可先讀。

## 常見面試題

1. **康威定律對微服務拆分的啟示是什麼？**（提示：服務邊界要對齊團隊擁有權；只拆程式不拆組織會得到什麼）
2. **什麼是反向康威操作？舉個你會怎麼用的例子。**（提示：先定目標架構、再調團隊；stream-aligned team 對齊業務能力）
3. **你接手的系統有明顯不合理的重複服務，怎麼判斷該合併還是保留？**（提示：先看組織——重複的根因若還在，技術上合併也會再長出來）
