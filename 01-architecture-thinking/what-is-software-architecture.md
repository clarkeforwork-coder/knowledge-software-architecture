# 什麼是軟體架構？架構師在決定什麼

## 前言

「我們專案的架構是 Spring Boot + MySQL + Redis。」——這句話你一定聽過，甚至說過。但它其實描述的是**技術選型清單**，不是架構。同一套技術棧，可以蓋出邊界清晰、十年還能改的系統，也可以蓋出誰都不敢動的一團泥球；差別不在清單，在清單之外的那些決策。

那架構到底是什麼？架構師（或扮演架構師角色的資深工程師）每天在決定的，究竟是哪些事？這篇筆記把「架構」從口號還原成可操作的定義——因為只有先說清楚「什麼是架構決策」，才談得上做好它。

## 技術背景

### 先破除誤解：架構不是技術選型，也不是那張框圖

牆上那張元件框圖、wiki 裡的技術棧清單，都只是架構的**投影**，不是架構本身。正式定義來自 ISO/IEC/IEEE 42010：架構是「系統在其環境中的基本概念或性質，體現於系統的元素、關係，以及設計與演進的原則之中」。這個定義嚴謹但抽象，實務上更好用的是兩個工程師給的版本：

- Ralph Johnson（由 Martin Fowler 在〈Who Needs an Architect?〉中引述）：架構是「專家開發者對系統設計的**共同理解**」，以及「那些**難以改變的決策**」——「架構就是重要的東西，不管那是什麼」。
- Grady Booch：「架構代表形塑系統的**重大設計決策**，而重大與否，以**改變的成本**衡量。」

三個定義收斂到同一件事：**架構是一組決策，篩選標準是改變的成本**。

### 什麼樣的決策算架構決策

用「改起來多貴」當尺，可以把日常決策排成一條光譜：

| 決策 | 影響範圍 | 改變成本 | 是架構決策嗎 |
|---|---|---|---|
| 服務間同步呼叫還是事件驅動 | 所有相關服務的失敗模式 | 全面翻修呼叫鏈 | ✅ 是 |
| 資料一致性採強一致還是最終一致 | 資料模型、補償邏輯、使用者體驗 | 幾乎等於重寫 | ✅ 是 |
| 模組邊界怎麼切、誰能呼叫誰 | 團隊分工、部署單位 | 跨團隊重構 | ✅ 是 |
| 用哪一套 JSON 序列化函式庫 | 封裝得當時只有一層 | 換掉一個 adapter | ❌ 通常不是 |
| 某支方法要不要抽小函式 | 單一檔案 | 一次 refactor | ❌ 不是 |

注意最後兩列的前提是「封裝得當」——**一個決策是不是架構級，本身就受架構影響**。JSON 函式庫若直接滲進每一層的方法簽名，換掉它就成了架構級工程。好的架構會主動把更多決策「降級」成可逆的小事，這也是它的價值所在。

### 架構師在決定什麼

把「架構＝難以改變的決策」展開，架構師的日常工作就是三件事：

1. **決定邊界**：系統切成哪些部分、每個部分對誰負責。邊界劃在哪，決定了未來每個需求要動幾個地方、過幾個團隊。
2. **決定契約**：邊界之間怎麼互動——同步或非同步、資料以什麼形狀流動、失敗時誰負責重試。（物件層級的契約見〈[分層物件家族](../02-architecture-styles/layered-object-family.md)〉。）
3. **決定品質屬性的優先序**：效能、可用性、可修改性不可能全拿，先輸哪個是**選擇**，不是意外——這是下一篇〈[品質屬性與取捨](quality-attributes-and-tradeoffs.md)〉的主題。

同樣重要的是「不在清單上」的事：架構師的產出不是框圖與文件，而是**決策與其理由**。圖會過期，理由才能讓後人判斷「當年的前提還成立嗎」。

## 實際案例

一個常見需求：「訂單成立後要通知會員系統發點數」。兩種寫法都能動：

```java
// 方案 A：同步呼叫——訂單服務直接呼叫會員服務
public void createOrder(Order order) {
    orderRepository.save(order);
    memberClient.addPoints(order.getMemberId(), order.calcPoints()); // 同步等待
}
```

```java
// 方案 B：事件驅動——訂單服務只宣告「事情發生了」
public void createOrder(Order order) {
    orderRepository.save(order);
    eventPublisher.publish(new OrderCreatedEvent(order.getId()));   // 發完即走
}
```

從程式碼看只差一行，但把「改變成本」的尺拿出來量，就會發現這是**光譜最左端**的決策：

- **失敗模式不同**：A 之下，會員系統掛掉會拖垮下單；B 之下，下單照常、點數延後補發。哪個可接受，取決於業務對「下單可用性」與「點數即時性」的優先序。
- **一致性語意不同**：A 是同一次請求內完成（但跨服務其實也不保證原子）；B 是最終一致，必須回答「事件丟了怎麼辦」「重複消費怎麼辦」。
- **擴散效應不同**：之後每加一個「訂單成立後要做的事」（發票、物流、風控），A 要繼續往下單流程裡加同步呼叫，B 只是多一個訂閱者。

選 A 還是 B 沒有標準答案——重點是**認出這是架構決策**：它該被拿到桌面上，用品質屬性討論、把理由記下來；而不是由當天寫這支方法的人順手決定。反例是真實世界最常見的架構事故來源：系統裡最重要的決策，往往是在沒人意識到「這是個決策」的情況下發生的。

## 技術優缺點

把「架構＝可辨識的決策集合」當工作方式，而不是把架構當文件或職稱：

**優勢**

- **討論有焦點**：爭論從「這樣設計好不好」變成「這個決策的改變成本多高、我們願意賭哪個前提」。
- **可傳承**：決策連同理由記錄下來（後續會談 ADR），新人能判斷歷史包袱哪些還有效。
- **授權清楚**：光譜右端的決策放手讓團隊自己做，架構師只看守昂貴的那一端，不會變成瓶頸。

**代價**

- **需要紀律**：辨識決策、記錄理由都是額外工夫，短期看不到產出。
- **可能過度上綱**：什麼都當架構決策來審，就是另一種形式的大前期設計（Big Design Up Front），拖慢所有人。

**這改變了什麼實務決策**：團隊需要一條明確的分界——什麼等級的決策要慢下來開會留紀錄，什麼等級直接做。「改變成本」就是那把尺；而尺要量得準，前提是說得出系統的品質屬性優先序。

## 小結

- 架構不是技術棧清單，也不是框圖；是**一組難以改變的決策**，以改變成本為篩選標準。
- 三個經典定義（ISO 42010、Ralph Johnson、Grady Booch）收斂於同一件事：重要、共識、改變昂貴。
- 架構師決定的是**邊界、契約、品質屬性優先序**；產出是決策與理由，不是圖。
- 好的架構會把更多決策降級成可逆的小事；最危險的架構決策，是沒人意識到它是決策的那些。
- 判斷「這是不是架構決策」的尺是改變成本；但要排品質屬性的優先序，得先把「快、穩、好維護」這些形容詞變成可以取捨的東西——下一篇：〈[品質屬性與取捨：沒有最好的架構，只有取捨](quality-attributes-and-tradeoffs.md)〉。

## 常見面試題

1. **架構與設計（design）的差別在哪？**（提示：不是位階高低，是改變成本與影響範圍的光譜；說得出「光譜」而非「二分」就到位了）
2. **你怎麼判斷一個技術決策需不需要拉架構討論？**（提示：改變成本、影響的團隊數、失敗模式是否外溢）
3. **「微服務」是架構嗎？**（提示：它是架構風格；套了風格不等於做了決策——邊界怎麼切、一致性怎麼給，才是決策本體）

## 延伸閱讀

- Martin Fowler, [Who Needs an Architect?](https://martinfowler.com/ieeeSoftware/whoNeedsArchitect.pdf)（IEEE Software, 2003）——Ralph Johnson 定義的出處，短而必讀
- ISO/IEC/IEEE 42010, *Systems and software engineering — Architecture description*——架構的正式定義
- Mark Richards & Neal Ford, *Fundamentals of Software Architecture*（O'Reilly, 2020）——第一章對「架構定義之難」的整理
- Frederick Brooks, *The Mythical Man-Month*——概念完整性（conceptual integrity）：共同理解為什麼重要
