# knowledge-software-architecture 專案規則

軟體架構技術筆記 repo（knowledge-* 家族的一員，portfolio 的一部分）。
筆記以繁體中文撰寫。慣例繼承自 [knowledge-java](../knowledge-java/CLAUDE.md)。

## 範圍（嚴格遵守）

- 只收 **軟體架構層級** 的主題：架構思維與品質屬性取捨、架構風格
  （分層、六角／Clean、事件驅動、單體 vs 微服務）、DDD（戰略與戰術設計）、
  分散式系統基礎（一致性、冪等性、分散式交易）、可靠性與擴展性模式
  （快取策略、訊息佇列模式、限流熔斷、back-pressure）、架構決策與文件（ADR、C4）。
- 特定產品的操作與調校（Redis 指令、Kafka 設定、Docker/K8s 部署）、
  語言與框架的實作細節**不收**，由其他 `knowledge-*` repo 承接。
  被要求寫這類主題時，先提醒範圍邊界。

## 與其他 knowledge-* repo 的分工

- 本 repo 談**模式與取捨**，不談實作細節：
  - Java 語言與 JVM 機制歸 [knowledge-java](../knowledge-java)。
    例：談分散式交易時，單機交易與資料庫鎖連回
    `../knowledge-java/08-data-access/transactions-and-database-locks.md`，不重寫基礎。
  - Spring 生態的實作歸 [knowledge-spring](../knowledge-spring)。
    例：談限流熔斷模式時，Spring 端實作以連結承接。
- 重疊主題**不重寫基礎**：筆記前言連回對應篇，內容從該篇的結尾接下去。
- 跨 repo 連結用相對路徑 `../knowledge-xxx/...`。

## 雙軌制

- 🔰 **基礎**（訓練主軌）：教學式筆記。每章以寫完 🔰 軌為「完整」。
- 🔬 **深入**（選修軌）：檔名以 `deep-` 開頭，寧缺勿濫，只寫最有價值的主題。
- 🔰 筆記結尾以「🔬 想深入：[標題](deep-xxx.md)」連到對應深入文（有才放）。

## 筆記結構

每篇筆記必須遵守 [TEMPLATE.md](TEMPLATE.md)，🔰 與 🔬 共用五段骨架：

- `## 前言`（工作中會遇到的提問切入）→ `## 技術背景`（先破除誤解再進主體，
  主體用 ### 展開）→ `## 實際案例` → `## 技術優缺點`（優勢 vs 代價）→
  `## 小結`（條列 3～5 條＋懸念鉤子連到 Roadmap 筆記）→
  附錄：`## 常見面試題`（1～3 題，只給方向提示）、
  `## 延伸閱讀`（經典書、論文、官方文件優先）。
- 兩軌差在「實際案例」的深度：🔰 是實務情境與示意演練；
  🔬 是論文／一手資料追讀、公開事故報告（post-mortem）分析、實驗模擬。
- 段落內規：先程式碼（或圖）後解釋、對比用表格、反模式用 ❌/✅ 註解。
- 圖片選配：放該章 `attachment/img/`、以內容命名（kebab-case）；
  概念圖優先 ASCII，AI 生圖當封面並檢查圖內亂字。

## 論述驗證（本 repo 以論述與圖解為主）

- 關鍵主張（數據、定理、案例）**必須標注可信出處**（經典書、論文、官方文件、
  原廠事故報告），不可憑印象編造；發現出處與記憶不符時，修正筆記內容而非引文。
- 少量示意程式碼以可讀為先；凡標示為**可執行**的程式碼與其輸出必須實測
  （JDK 17 Temurin，`java Xxx.java` 單檔執行）。
- 🔬 筆記引用論文或事故報告時，標注版本／發表年份與章節。

## 可執行範例（混合制）

- 預設文內 snippet 與 ASCII 圖即可，原則上**不**附範例檔。
- 只有「不跑看不出結果」的主題（重試風暴、快取雪崩、競態在分散式情境的模擬）
  才附範例檔：放該章 `examples/` 子目錄、**單一 .java 檔**、
  可用 `java Xxx.java` 直接執行，不引入 Maven/Gradle。

## README 是索引與進度表

- 新增筆記時**必須**同步更新 README 目錄表（深度、狀態兩欄），
  並把規劃中的標題換成實際連結。
- 狀態：✅ 完成（符合模板）/ 🔧 待翻新（內容完成、尚未符合模板）/
  🚧 進行中 / 📝 待補。
- 翻新一篇 🔧 筆記使其符合模板後，把狀態改回 ✅。
- 章節資料夾（`NN-kebab-name/`）在該章第一篇筆記落地時才建立，不預建空資料夾。
- 寫作優先順序依 README 的 Roadmap（近期 → 中期 → 長期）。
