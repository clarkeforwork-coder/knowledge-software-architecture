# 戰略設計：Bounded Context 與 Context Map

## 前言

一場跨部門會議，為了「商品」這個詞吵了三小時：庫存部門的商品有儲位和批號、行銷部門的商品有文案和活動價、財務部門的商品是一個會計科目。最後有人拍板：「建一張統一的商品主檔，欄位大家都加進去。」——恭喜，一個八十個欄位、誰改誰痛的怪物就此誕生。

上一章四種架構風格談完，收在同一個未解的字：**邊界**。層的邊界、服務的邊界，到底該沿著什麼劃？DDD 戰略設計的答案簡潔得出乎意料：**沿著語言的斷層劃**。這篇筆記講清楚 Bounded Context 是什麼、為什麼大一統模型必然失敗、以及 Context Map 怎麼把邊界之間的關係攤在桌面上。

## 技術背景

### 先破除誤解：用了 Repository 不等於在做 DDD

多數自稱 DDD 的專案，其實只是採用了 Entity、Repository、Service 這套**戰術樣式**——社群戲稱 DDD-Lite。但《Domain-Driven Design》（Eric Evans, 2003）的重心在後半部：**戰略設計先於戰術設計**。邊界劃錯了，邊界內的程式碼再精緻都是白工；反過來，戰略邊界劃對了，邊界內就算寫的是樸素的交易腳本，系統仍然是健康的。

### 統一語言：模型和話語必須是同一套

DDD 的起手式是統一語言（Ubiquitous Language）：**團隊口中的話、文件裡的詞、程式碼裡的類別名，必須是同一套詞彙**。業務說「訂單成立」，程式碼就該有 `placeOrder`，而不是 `insertOrderRecord`——詞彙的落差就是理解的落差，理解的落差遲早變成 bug。

但統一語言有個天生的限制：**它只能在一個範圍內統一**。「商品」對庫存和行銷本來就是兩個概念，強求全公司統一，得到的不是共識而是三小時的會議。

### Bounded Context：模型的國境線

Bounded Context（限界上下文）就是那個「範圍」：**一個模型、一套語言能夠自洽適用的邊界**。邊界之內，「商品」有唯一且精確的意義；跨過邊界，同一個詞可以、也應該是另一個模型：

- 庫存上下文的「商品」：儲位、批號、安全庫存——關心「東西在哪、還剩多少」。
- 行銷上下文的「商品」：文案、圖片、活動價——關心「怎麼讓人想買」。
- 財務上下文的「商品」：科目、稅別、成本——關心「怎麼入帳」。

三個模型各自薄而準。大一統模型的失敗不是執行不力，是**結構必然**：把 N 個部門對同一名詞的關注面塞進一個模型，每個使用者都只用到一小片、卻要承受全部的變更；語意衝突（行銷的「上架」和庫存的「上架」時點不同）則被欄位註解和 if-else 掩埋，直到出帳那天爆炸。

### Context Map：把邊界之間的關係畫出來

邊界劃出來後，上下文之間的**權力關係**必須顯性化，這就是 Context Map（Evans 在書中第 14 章給出的模式清單）。實務最常用的幾種：

| 關係模式 | 意思 | 風險訊號 |
|---|---|---|
| 客戶—供應商（Customer-Supplier） | 下游需求會進上游的排程 | 上游排程失能時退化成下一列 |
| 順從者（Conformist） | 下游直接採用上游的模型，不翻譯 | 省整合成本，代價是模型主權讓渡 |
| 防腐層（Anticorruption Layer） | 下游築翻譯層保護自己的模型 | 翻譯層本身的維護成本 |
| 共享核心（Shared Kernel） | 兩個上下文共享一小塊模型 | 共享區的每次修改都要雙方同意 |
| 各行其道（Separate Ways） | 乾脆不整合，各做各的 | 資料重複；但常比爛整合便宜 |

Context Map 的價值不在圖多好看，在**逼問**：這條整合線上，誰遷就誰？答案不寫下來，預設答案就是「下游工程師在深夜遷就」。

### 邊界的三重對齊

Bounded Context 是三個世界的交會點：它是**模型**的邊界（DDD）、最好也是**團隊**的邊界（〈[康威定律](../01-architecture-thinking/conways-law.md)〉——一個上下文一個歸屬團隊，語言才統一得起來）、還常是**部署**的邊界（〈[單體 vs 微服務](../02-architecture-styles/monolith-vs-microservices.md)〉的「沿驗證過的縫下刀」，那條縫最好就是上下文邊界）。三者對齊時系統最順；三者錯位時，康威定律會慢慢把另外兩個拖向組織圖的形狀。

## 實際案例

一家電商的商品主檔演化史，兩條路線推演：

**路線一：大一統**。`PRODUCT` 表八十個欄位，庫存、行銷、財務欄位混居。行銷要加「限時價開始時間」——DBA 排程兩週，因為要全公司回歸測試；庫存改「批號規則」——行銷的活動頁莫名壞掉，因為有人共用了那個欄位存別的東西；最傳神的症狀是欄位名：`STATUS`、`STATUS2`、`REAL_STATUS`——三個狀態欄，沒人敢刪前兩個。每個詞的意義都要靠部落知識還原，新人問「這欄位是幹嘛的」，答案以「看情況」開頭。

**路線二：沿語言斷層切開**。三個上下文各自建模：庫存的 `StockItem`、行銷的 `Listing`、財務的 `AccountingItem`——注意連名字都不再是同一個詞，因為它們本來就不是同一個概念，「商品」只是日常口語的方便說法。行銷加欄位只動行銷的模型與表；三者之間用 Context Map 講清楚：行銷是庫存的下游（顯示「還剩多少」用，Conformist 即可，數字晚幾秒無妨）；財務對兩者都是下游但模型必須自主（成本認列規則不能被上游改動牽走——這裡值得一道防腐層，見本章第三篇）。

代價也如實發生：三個上下文都存了「商品名稱」的複本，同步靠事件（〈[事件驅動架構](../02-architecture-styles/event-driven-architecture.md)〉的正面用例）；偶爾有短暫不一致，但換來的是**三年沒有再開過一次「商品是什麼」的會**。

## 技術優缺點

**優勢**

- **模型薄而準**：每個上下文只為自己的問題建模，認知負荷與變更爆炸半徑同時縮小。
- **語意衝突顯性化**：「同詞不同義」從埋在欄位裡的地雷，變成邊界上的明文翻譯。
- **整合關係可治理**：Context Map 把「誰遷就誰」從默契變成決策。

**代價**

- **資料重複與最終一致**：跨上下文的複本需要同步機制與業務端的理解。
- **整合成本前置**：邊界之間的每條線都要選關係模式、建轉換，不再有「直接 join 那張表」的捷徑。
- **劃錯要重劃**：語言斷層的判讀需要領域知識，初版邊界常有誤差；重劃上下文比重構一個類別貴得多。

**這改變了什麼實務決策**：拆分討論的第一個問題從「這個服務會不會太大」變成「**這裡面有幾套語言**」。偵測訊號很日常：同一個詞在兩群人嘴裡意思不同、同一張表有兩群人在搶欄位、同一個狀態機有人想加第九種狀態——都是語言斷層的位置，也就是邊界候選的位置。

## 小結

- DDD 的重心在戰略不在戰術：邊界劃錯，邊界內再精緻都是白工。
- 統一語言只能在一個範圍內統一；那個範圍就是 Bounded Context——一個模型自洽適用的國境線。
- 大一統模型的失敗是結構必然：N 個關注面塞一個模型，人人只用一片、人人承受全部。
- Context Map 逼問每條整合線「誰遷就誰」；不回答，答案就是下游工程師在深夜遷就。
- 上下文、團隊、部署單位三重對齊時系統最順——邊界的學問跨過了組織與架構風格兩章，在此合流。
- 邊界之內呢？模型自己要怎麼長——什麼是 Entity、什麼是 Value Object、一致性的最小單位在哪？下一篇：〈[戰術設計：Aggregate、Entity 與 Value Object](tactical-design-aggregate.md)〉。

## 常見面試題

1. **Bounded Context 和微服務是一對一的關係嗎？**（提示：上下文是模型邊界、服務是部署邊界；理想對齊但不必然——一個上下文拆多個服務、多個上下文住同一個模組化單體都成立）
2. **兩個上下文都需要「會員」資料，你會怎麼設計？**（提示：先問兩邊的「會員」是不是同一個概念；再談複本＋事件同步 vs 上下游關係的選擇）
3. **怎麼發現邊界劃錯了？**（提示：跨上下文的修改總是成對出現、翻譯層淪為透傳、兩個團隊天天開整合會——邊界沒有落在語言斷層上的症狀）
