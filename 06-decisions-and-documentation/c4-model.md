# C4 模型：畫給人看的架構圖

## 前言

會議白板上的「架構圖」：有人畫的框是伺服器、有人畫的是模組、有人畫的是團隊；箭頭沒有標籤，沒人知道那條線是呼叫、部署還是「有點關係」。十個工程師畫同一個系統，得到十張層級混雜的圖——然後大家對著圖各說各話，以為達成了共識。

問題不是大家不會畫圖，是**圖沒有比例尺**。地圖學早就解決了這件事：世界地圖不標便利商店，街道圖不畫大洲——每張圖有自己的縮放層級與受眾。C4 模型（Simon Brown）做的就是把「比例尺」帶進架構圖。這是本書最後一篇：決策的「為什麼」交給了 [ADR](adr.md)，系統的「長什麼樣」由這篇收尾。

## 技術背景

### 先破除誤解：不存在「一張圖說明一切」

想把系統全貌塞進一張圖的結果，是每個層級都出現一點、每個層級都講不清——**一張圖有所有層級，等於沒有層級**。C4 的核心動作是拆層：像地圖縮放一樣，四個固定的觀察高度，每張圖只站在一個高度上。

### 四層：Context、Container、Component、Code

| 層 | 框代表什麼 | 回答的問題 | 受眾 |
|---|---|---|---|
| L1 系統情境（Context） | 你的系統（一整個框）＋使用者＋外部系統 | 這系統為誰服務、跟誰往來？ | 所有人，含非技術方 |
| L2 容器（Container） | 可獨立部署／執行的單元：應用、資料庫、佇列 | 系統由哪幾塊跑起來的東西組成？ | 工程團隊、維運 |
| L3 元件（Component） | 單一容器內部的模組與職責 | 這個應用裡面怎麼分工？ | 該容器的開發者 |
| L4 程式碼（Code） | 類別圖 | 幾乎不畫——IDE 隨時能生成 | 沒有人需要維護它 |

（注意 Container 指「執行單元」，與容器化技術無關——命名撞車純屬歷史意外。）

**八成的溝通價值在前兩層**：Context 圖是跟業務、新人、外部團隊對話的共同地面；Container 圖是工程內部的日常地圖。層級往下，圖的受眾越窄、腐爛越快——這直接決定了維護策略（見後）。

### 圖的紀律：讓圖自己會說話

C4 一半的價值不在四層，在它順手帶進來的畫圖衛生習慣：

- **每個框**：名字＋類型＋一句話職責——「訂單服務／後端應用／接單與訂單生命週期管理」，不是一個孤零零的「Order」。
- **每條線**：有方向、有動詞標籤——「查詢庫存（同步 HTTP）」，不是一條裸箭頭。第二章那個「資料庫為什麼呼叫佇列？」的白板懸案，九成是無標籤箭頭的冤獄。
- **一張圖一個層級**：Container 圖裡出現某個服務的內部類別，就是混層的開始。
- **有圖例**：框的顏色與線的虛實一旦有含義，就要寫出來。

### 圖與文字的分工，以及怎麼不過期

C4 呈現**結構**（what），ADR 呈現**理由**（why）——Container 圖上那條「直接讀複本」的線，旁註 `ADR-0017`，兩份文件互相引用，各司其職。

圖會過期的問題，答案有兩半：其一是**只承諾維護值得維護的層級**——L1、L2 變動慢、受眾廣，納入維護；L3 按需畫、用完即棄，承認它短命。其二是 **diagrams as code**——用文字描述生成圖（Structurizr DSL、Mermaid 等），圖進 repo、隨 PR diff、跟程式碼同步演進；本 repo 通篇的 ASCII 圖是同一哲學的極簡版：能 diff 的圖才活得下來。

## 實際案例

把本書從第一章用到現在的那個電商系統，畫成前兩層（ASCII 示意，實務工具會更豐富）：

**L1 Context——給所有人看的那張**：

```
   [顧客]           [客服人員]
      │ 瀏覽、下單        │ 查單、退款
      v                  v
┌─────────────────────────────┐
│        電商平台（本系統）      │
└──┬──────────┬──────────┬────┘
   │ 扣款請求   │ 商品主檔同步 │ 發送通知
   v          v          v
[金流商]    [遺留 ERP]   [簡訊/Email 商]
```

**L2 Container——工程團隊的日常地圖**：

```
[顧客瀏覽器] ──HTTPS──> [Web 前端 SPA]
                            │ REST
                            v
                     [API 閘道] ──限流/認證
                       │
        ┌──────────────┼────────────────┐
        v              v                v
  [訂單服務]      [商品目錄服務]      [通知服務]
   │    │  發布 OrderPlaced │ ACL 讀 ERP   ^
   │    └────> [訊息佇列] ──────訂閱───────┘
   v                   
[訂單資料庫]   [商品快取]─[商品資料庫]
（ADR-0017：列表查詢直讀複本）
```

**同一個系統、不同的對話**：跟業務方討論「要不要換簡訊商」，開 L1——三十秒對齊範圍，不需要解釋什麼是佇列；新工程師報到，給 L2——十五分鐘知道有幾個可部署單元、事件從哪裡流到哪裡、哪條線掛著 ADR 編號；只有要動訂單服務內部（聚合怎麼切、防腐層在哪個套件）才畫一張短命的 L3。反過來的災難也常見：對業務方展示 L2 級細節，換來一小時的「所以佇列是什麼」；工程評審只有 L1，等於拿世界地圖討論巷口動線。

## 技術優缺點

**優勢**

- **比例尺即共識**：開會先問「我們在看第幾層」，十張各說各話的白板圖收斂成一套座標系。
- **受眾對齊**：每層有明確的讀者，不再用同一張圖同時得罪業務與工程。
- **可版控**：diagrams as code 讓圖進 repo、可 diff、可 review——圖的變更成為 PR 的一部分。
- **onboarding 提速**：L1＋L2＋ADR 目錄，是新人第一週最高密度的三份材料。

**代價**

- **維護成本不會歸零**：承諾維護的 L1／L2 仍會漂移，需要「動架構的 PR 要帶圖」的紀律。
- **不表達動態行為**：C4 是結構快照；「下單之後發生什麼」的時序需要另配序列圖，事件驅動系統尤其如此（〈[事件驅動架構](../02-architecture-styles/event-driven-architecture.md)〉的因果鏈不可見問題，靜態圖救不了）。
- **不表達部署拓撲**：機房、副本數、網段是部署圖的事——C4 有對應的補充圖型，但別硬塞進四層。

**這改變了什麼實務決策**：畫圖前先問兩個問題——「**這張圖給誰看？他站在哪個高度？**」；repo 的文件目錄從此有標準配置：`docs/adr/`（為什麼）＋ L1／L2 圖（長什麼樣），兩者互相引用，構成系統的最小可信文件集。

## 小結

- 架構圖的病根是沒有比例尺；C4 用四個固定高度（Context／Container／Component／Code）給圖裝上縮放。
- 八成價值在前兩層：L1 對所有人、L2 對工程團隊；L3 短命按需、L4 不畫。
- 圖的衛生習慣與層級同等重要：框有職責、線有動詞、一圖一層、附圖例。
- C4 管 what、ADR 管 why，互相引用；能進 repo、能 diff 的圖才活得下來。
- 全書至此收束，六章其實是一條線：用**改變成本**辨認架構決策（01）→ 在風格光譜上取捨（02）→ 沿語言斷層劃邊界（03）→ 付分散式的必答稅（04）→ 讓系統在過載與故障下活著（05）→ 把「為什麼」與「長什麼樣」留給後人（06）。架構沒有終點——但有紀錄的架構，至少讓下一個人不必從考古開始。

## 常見面試題

1. **C4 的四層分別是什麼、給誰看？**（提示：Context／Container／Component／Code；受眾遞窄、腐爛遞快、L4 不畫）
2. **一張「好」的架構圖有哪些基本紀律？**（提示：一圖一層、框有職責、線有方向與動詞、圖例；反例就是白板上那張）
3. **架構圖怎麼避免過期？**（提示：只維護 L1/L2、diagrams as code 進 repo 隨 PR diff、動架構的 PR 帶圖）
