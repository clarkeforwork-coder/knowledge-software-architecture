# ADR：把「為什麼」留下來

## 前言

新人指著程式碼問：「訂單查詢為什麼繞過服務層直接讀複本？這不是違反我們自己的分層規範嗎？」資深工程師的回答是這個行業最常見的一句話：「**不知道，前人弄的，動了怕出事。**」

系統不會失憶——程式碼都在；失憶的是**理由**。第一章說過，架構師的產出是「決策與其理由」，而不是圖（〈[什麼是軟體架構](../01-architecture-thinking/what-is-software-architecture.md)〉）——本書走到最後一章，來兌現那句話的下半句：理由要怎麼留下來，才不會跟著人員流動一起蒸發。答案小得出奇：一種一到兩頁的短文件，Architecture Decision Record。

## 技術背景

### 先破除誤解：架構文件的失敗不是因為沒人寫，是因為寫錯了本體

多數團隊試過「架構設計書」：五十頁、有目錄、有版本號，寫完的那天就是它最準確的一天——之後系統持續演進，文件持續過期，半年後沒人敢信它，一年後沒人記得它在哪。快照式大文件的失敗是**結構性**的：它記錄的本體是「現狀」，而現狀天生會變。

ADR（Michael Nygard，2011 年〈Documenting Architecture Decisions〉）換了一個不會過期的本體：**決策**。「2024 年 3 月，我們在那樣的約束下，做了這個選擇」——這是歷史事實，永遠為真；之後可以被新決策**取代**（supersede），但不會「過期」。文件從一張會腐爛的快照，變成一條只增不改的決策日誌。

### 骨架：五個欄位

一篇 ADR 一到兩頁、一個決策、存在 repo 裡與程式碼同版本演進：

| 欄位 | 寫什麼 | 要點 |
|---|---|---|
| 標題 | 編號＋一句話決策 | 「ADR-0017：訂單查詢讀複本」 |
| 狀態 | proposed → accepted → superseded by ADR-NNNN | 狀態機，不刪不改 |
| 情境（Context） | 當時的力場：問題、約束、擺在桌上的選項 | 寫「當時」，讓後人能判斷前提還在不在 |
| 決策（Decision） | 我們選了什麼 | 主動語態、一段話 |
| 後果（Consequences） | 這個選擇**買到什麼、付出什麼** | 誠實寫代價——只寫好處的 ADR 是宣傳品 |

Consequences 是整個格式的靈魂，也正是本書每篇筆記「技術優缺點」一節的同款紀律：**沒有寫下代價的決策，等於沒有被完整地做過**。

### 生命週期：不可變，只能被取代

改變主意時**不改舊 ADR**——寫一篇新的，狀態欄標 `supersedes ADR-0017`，舊篇標 `superseded by ADR-0031`。於是決策鏈本身成為系統的演化史：讀完一條 supersede 鏈，你知道的不只是「現在怎麼做」，還有「我們曾經怎麼做、為什麼變了」——這是任何快照文件給不了的東西。

### 哪些決策值得寫

用第一章那把尺：**改變成本**。難以改變的決策（風格選擇、一致性策略、邊界劃分、對外契約）值得一篇 ADR；換個函式庫、抽個方法不值得——把雞毛蒜皮也寫成 ADR，訊號就被稀釋成噪音。實務的流程輕得很：ADR 以 PR 提交，code review 就是決策評審；`docs/adr/` 目錄按編號排隊，新人報到第一天从 0001 讀起。

## 實際案例

回到開頭那個「繞過服務層讀複本」的謎題。有 ADR 的平行宇宙裡，新人搜尋目錄，三分鐘後讀到：

```markdown
# ADR-0017：訂單列表查詢直接讀取資料庫複本

狀態：accepted（2024-03-12）→ 部分由 ADR-0021 補強（2024-06-02）

## 情境
促銷期間報表與列表查詢多次打滿主庫連線池，拖垮下單寫入（見 2024-02 事故報告）。
選項：(a) 查詢全走服務層並加快取；(b) 列表/報表類唯讀查詢直接讀複本；
(c) 讀寫分離中介層。當時團隊 4 人，(c) 的建置成本不可行；(a) 的快取失效
複雜度對「列表頁」收益過低。

## 決策
唯讀的列表與報表查詢繞過服務層，直接讀複本；寫入與單筆詳情維持原路徑。
此為分層規範的顯式例外，範圍限 query 套件。

## 後果
- 買到：主庫寫入與查詢隔離，促銷期下單不再受報表牽連。
- 付出：複寫延遲下，使用者剛下的單可能不在列表上（read-your-writes 缺失）
  ——上線後客訴驗證了這點，由 ADR-0021（寫後讀主 30 秒）補上。
- 付出：分層規範出現一個需要守衛的例外；架構測試已將豁免範圍限定在 query 套件。
```

三分鐘裡發生的事，值得逐一點名：新人得到的不是「答案」而是**力場**——他能判斷「當年 4 人團隊」的前提如今（12 人、已有讀寫分離中介）還成立嗎？不成立，他可以理直氣壯地提 ADR-0035 取代它，而不是「動了怕出事」；那條 `→ ADR-0021` 的鏈告訴他這個決策踩過什麼坑、怎麼補的（正是〈[一致性模型](../04-distributed-basics/consistency-models.md)〉的教科書劇情）；而「顯式例外＋架構測試守衛」讓這個繞道不會被複製貼上成第二、第三個無主例外。**同一個決策，有沒有這一頁紙，差別是「可以被重新評估的資產」與「不敢碰的祖產」。**

## 技術優缺點

**優勢**

- **理由可傳承**：人走了，力場留著；「為什麼」不再依附於某個人的記憶。
- **重新評估有依據**：Context 讓後人檢查前提，決策從聖旨變成可挑戰的假設。
- **評審有焦點**：ADR 的 PR 逼討論收斂到「情境—選項—後果」，而不是會議室裡的自由搏擊。
- **新人加速**：按編號讀一遍決策鏈，比讀十份過期設計書更接近系統的真相。

**代價**

- **紀律成本**：決策當下多花三十分鐘寫作——最忙的時刻付費，收益在未來，天然反直覺。
- **會退化**：status 沒人維護、supersede 鏈斷掉，ADR 目錄就變成另一種過期文件——它降低了腐爛速度，沒有豁免腐爛。
- **訊號稀釋的風險**：門檻失守、什麼都寫，重要決策淹沒在瑣事裡。

**這改變了什麼實務決策**：架構討論的收尾動作從「散會」變成「**誰來提這篇 ADR 的 PR**」；而評審一個有爭議的設計時，第一個要求從「畫個圖來看看」變成「把 Context 和被否掉的選項寫出來」——很多爭論在被迫寫下選項的那一刻自動終結。

## 小結

- 快照式文件的失敗是結構性的：本體是「現狀」就必然過期；ADR 把本體換成「決策」——歷史事實不過期，只被取代。
- 五欄骨架，靈魂在 Consequences：不寫代價的決策等於沒被完整做過。
- 不可變＋supersede 鏈＝系統的演化史；「為什麼變了」與「現在是什麼」同樣重要。
- 篩選的尺與架構決策同一把：改變成本；流程輕量化——ADR 進 repo、PR 即評審。
- ADR 把決策的「為什麼」留下來了；但系統的「長什麼樣」——結構本身——要怎麼講給不同的人聽？最後一篇：〈[C4 模型：畫給人看的架構圖](c4-model.md)〉。

## 常見面試題

1. **ADR 和傳統設計文件差在哪？為什麼它比較不容易過期？**（提示：本體是決策不是現狀；不可變＋supersede）
2. **決策後來被推翻了，ADR 怎麼處理？**（提示：不改舊篇，新篇 supersede；決策鏈即演化史）
3. **什麼決策值得寫 ADR？誰決定？**（提示：改變成本那把尺；PR 流程讓團隊而非個人把關）
