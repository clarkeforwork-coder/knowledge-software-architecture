# 筆記模板

與 [knowledge-java](../knowledge-java/TEMPLATE.md) 同一套骨架。
🔰 與 🔬 **共用同一套五段骨架**，差別在深度：
🔬 的「實際案例」是論文／一手資料追讀、公開事故報告分析或實驗模擬；
🔰 的是實務情境與示意演練。

## 五段骨架

```markdown
# 標題

（封面圖，選配）

## 前言

用一個工作中會遇到的提問切入（「為什麼系統拆成微服務之後，反而變慢了？」），
說明為什麼值得懂。2～3 段內。
與 knowledge-java／knowledge-spring 有對應篇時，在此連回：「本文從〈xxx〉的結尾接下去」。

## 技術背景

先破除一個常見誤解、或鋪必要的前置概念，再進入主體。
主體用 ### 子章節展開。

## 實際案例

一段完整的情境推演或案例分析。
🔰：實務情境、架構演進的決策過程、真實的踩坑後果。
🔬：論文／一手資料追讀、公開事故報告（post-mortem）分析、實驗模擬——
呈現「怎麼查到的」，不只是結論。

## 技術優缺點

這個設計的優勢 vs 代價（取捨），收在「這改變了什麼實務決策」。

## 小結

- 條列回顧 3～5 條帶走訊息
- 結尾放懸念鉤子，連到 Roadmap 上的下一篇：
  已完成的直接連結；未完成的寫「見規劃中的〈標題〉」，該篇完成後回頭補連結。
  🔰 有對應深入文時，鉤子連到 `deep-` 筆記。

## 常見面試題

1～3 題經典問法（系統設計面試常客），附一句話方向提示（不寫完整解答，答案就在上文）。

## 延伸閱讀

- 經典書（DDIA、Clean Architecture、DDD）、論文、官方文件、
  原廠事故報告優先；避免易失效的部落格連結
```

## 段落內規（骨架之下的寫法）

- **先程式碼（或圖）、後解釋**——範例與圖解在前，文字說明在後
- 對比類主題用表格（如 Orchestration vs Choreography）
- 反模式用 ❌/✅ 標注在程式碼註解或圖說裡
- 示意程式碼以可讀為先；凡標示為可執行者，必須真的能跑（並經實測）

## 圖片慣例

- 圖片**選配**，不強制。
- 放該章 `attachment/img/`，以圖片內容命名（kebab-case），
  例如 `saga-orchestration-flow.png`。
- 概念圖、流程圖**優先用 ASCII**（好 diff、不會有亂字）；
  AI 生圖適合當封面與氛圍圖，使用前檢查圖內文字是否有亂碼。
- 引用格式：`![內容描述](attachment/img/xxx.png)`。

## 可執行範例（混合制）

- 本 repo 以論述與圖解為主，預設文內 snippet 與 ASCII 圖即可，**不**另附範例檔。
- 只有「不跑看不出結果」的主題才附：重試風暴、快取雪崩、
  競態在分散式情境的模擬等。
- 附檔放在該章的 `examples/` 子目錄，例如 `05-reliability/examples/RetryStormDemo.java`。
- 每個範例是**單一 .java 檔**，用 `java RetryStormDemo.java`（JDK 11+ 單檔執行）
  就能跑，不引入建置工具。
- 筆記內文用一行連過去：「▶ 可執行範例：[RetryStormDemo.java](examples/RetryStormDemo.java)」。
