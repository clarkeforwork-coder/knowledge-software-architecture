# 單體 vs 微服務：什麼時候該拆

## 前言

「我們要不要上微服務？」——這個問題出現的場合通常不太對勁：老闆讀了一篇大廠技術文、隔壁團隊在會議上簡報了服務網格、或者面試官把它當送分題。很少是因為有人拿著監控數據說「這兩個模組的擴縮需求差了十倍」。

這篇筆記想把問題修正成可以回答的形狀。「單體 vs 微服務」不是落後與先進，不是遲早要走的進化路線，而是一組**用分散式複雜度購買組織與部署自由**的交易。買之前，先看清楚商品和價格。

## 技術背景

### 先破除誤解：微服務不是單體的進化型

微服務（Lewis & Fowler 對此風格的界定見〈Microservices〉, 2014）不在單體的「下一階段」，兩者是不同取捨下的平行選項。連「先做哪個」都沒有共識——Fowler 主張〈Monolith First〉（2015）：幾乎所有成功的微服務系統都從單體長出來，因為邊界只有在運行中的系統裡才看得清；Stefan Tilkov 同年反論〈Don't start with a monolith〉：如果一開始就確定要拆，單體養出的緊耦合會讓日後的拆分更痛。兩篇對讀，比背任何一篇都有價值——他們吵的其實是同一件事：**你對邊界的把握有多少**。

### 你在買什麼

微服務的商品清單，每一項都真實存在：

- **獨立部署**：一個服務一列部署列車，改點數邏輯不用等訂單團隊的測試。
- **獨立擴縮**：搜尋服務吃 CPU、圖片服務吃頻寬，各自加各自的機器。
- **故障隔離**：推薦服務死透，下單照常（前提是你做對了隔離）。
- **團隊自治**：服務即所有權邊界，正面運用〈[康威定律](../01-architecture-thinking/conways-law.md)〉。
- **技術異質**：個別服務可以換語言換儲存（這項最常被高估——用到的團隊遠比想像少）。

### 你在付什麼

價格是**分散式系統的必然稅**，一毛都逃不掉：

- 行程內呼叫變網路呼叫：延遲、逾時、重試、部分失敗——單體裡不存在的整類失敗模式。
- 跨服務資料一致性：交易邊界被拆開，最終一致與補償設計成為日常（見規劃中的〈分散式交易：2PC、Saga 與 Outbox〉）。
- 運維地基：服務發現、集中日誌、分散式追蹤、部署管線 × N——上一篇〈[事件驅動架構](event-driven-architecture.md)〉帳單的加強版。
- 本機開發與整合測試：起十五個服務才能跑一條流程的痛。

### 被跳過的中間解：模組化單體

多數「單體的痛」其實是**模組化失敗**的痛：模組間依賴亂竄、共享資料表誰都能寫、改一處全案回歸。這些病拆行程治不好——把爛邊界拆成網路呼叫，得到的是〈[康威定律](../01-architecture-thinking/conways-law.md)〉案例裡的分散式巨石：物理上分開、邏輯上仍要一起改一起上。

模組化單體（modular monolith）把順序排對：**先在單一部署單元內立起模組邊界**——模組間只能走明定的介面、各管各的資料表、依賴方向用架構測試守住（手段同〈[分層架構與它的邊界](layered-architecture-and-boundaries.md)〉，方向從層間改成模組間）。邊界在單體內被驗證過，日後要拆，沿著驗證過的縫下刀；不需要拆，就繼續享受單一部署的簡單。

### 該拆的訊號

拆分決策應該由**具體的痛**驅動，每一項都能對回品質屬性場景（〈[品質屬性與取捨](../01-architecture-thinking/quality-attributes-and-tradeoffs.md)〉的方法在此複用）：

1. 部署列車互相卡死：一個模組的 hotfix 要等全案回歸。
2. 擴縮需求差異大到浪費明顯：為了 5% 的熱點模組整體加倍機器。
3. 團隊規模超過溝通結構負荷：多團隊在同一程式庫互踩，合併衝突與發版協調成為常態。
4. 邊界已在單體內驗證：模組介面穩定、資料自持——這條是前三條的前提，不是第四個選項。

## 實際案例

### 故事一：履歷驅動的三十個服務

五人新創，產品還在找方向，技術負責人決定「一步到位」：三十個微服務、服務網格、每服務獨立資料庫。六個月後的體檢：五個人裡兩個全職在餵基礎設施；一個「訂單改欄位」需求跨四個服務、四個 repo、四條部署管線；本機起環境要寫一頁 wiki；而產品方向一轉，三十個邊界裡有二十個劃錯了位置——**邊界劃在產品定型之前，等於把猜測固化成網路契約**。最後全員停工兩個月，把服務合併回三個。他們付了微服務的全額稅金，卻沒有一項商品用得上：沒有需要獨立擴縮的負載、沒有需要自治的多團隊、沒有互相卡死的部署列車。

### 故事二：沿著驗證過的縫下刀

另一家公司，單體跑了四年，但立了規矩：模組間走介面、資料表各自持有、架構測試守依賴。金流模組後來出現兩個具體訊號——合規要求它獨立稽核與隔離部署；大促期間它的負載曲線與其他模組完全不同步。於是拆分：介面早已存在（改成 API 呼叫）、資料表本來就沒人跨界讀寫（直接搬走）、呼叫方語意不變。兩週完成，其餘模組繼續留在單體裡——**拆分是外科手術，不是宗教改宗**：哪裡有病灶開哪裡，健康的器官不動。

兩個故事的差別不在技術能力，在**順序**：邊界的驗證發生在拆分之前還是之後。

## 技術優缺點

本篇的主體就是取捨，這裡收斂成決策視角：

| 情境 | 合理預設 |
|---|---|
| 產品未定型、團隊 < 10 人 | 模組化單體——邊界還在流動，別把猜測固化成網路契約 |
| 單體病灶是模組邊界糊掉 | 先修邊界（模組化），拆行程治不了這個病 |
| 特定模組有獨立擴縮／合規／發版節奏需求，且邊界已驗證 | 沿縫拆出該模組，其餘不動 |
| 多團隊互踩、部署互卡、邊界穩定 | 微服務的商品此時才對症，逐步拆 |

**這改變了什麼實務決策**：預設答案是模組化單體；每一次拆分由品質屬性場景與具體病灶驅動，而不是由技術潮流或履歷驅動。「我們要不要上微服務」的正確回法是反問：**你現在最痛的是哪一條？拆行程治得了它嗎？**

## 小結

- 單體與微服務是平行的取捨選項，不是進化階段；Monolith First 與其反論吵的核心是「你對邊界的把握」。
- 微服務的商品是獨立部署、獨立擴縮、故障隔離、團隊自治；價格是分散式稅——網路失敗模式、最終一致、運維地基。
- 多數單體之痛是模組化失敗，拆行程治不好爛邊界，只會得到分散式巨石。
- 模組化單體是被跳過的正解：先在行程內驗證邊界，日後沿驗證過的縫下刀。
- 拆分由具體病灶與品質屬性驅動，是外科手術不是改宗。
- 本章四種風格談完了，但每一篇都繞著同一個未解的字：**邊界**。層的邊界、圓心的邊界、服務的邊界——它們到底該沿著什麼劃？答案不在技術裡，在業務裡。下一章進入 DDD：〈[戰略設計：Bounded Context 與 Context Map](../03-ddd/strategic-design-bounded-context.md)〉。

## 常見面試題

1. **微服務解決什麼問題？不解決什麼問題？**（提示：商品清單 vs 「模組化失敗拆了也沒用」；說得出分散式巨石就到位）
2. **你怎麼判斷系統該開始拆了？**（提示：部署互卡、擴縮差異、團隊互踩三訊號＋邊界已驗證這個前提）
3. **什麼是模組化單體？模組邊界靠什麼守住？**（提示：介面、資料自持、架構測試——和分層守邊是同一套機制，換了方向）
