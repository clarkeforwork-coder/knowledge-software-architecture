# 事件驅動架構：解耦的代價

## 前言

「訂單服務憑什麼要知道會員系統、發票系統、物流系統的存在？發個事件出去，誰關心誰訂閱。」——這是事件驅動最迷人的一句話，而且它是真的。三個月後的另一句話也是真的：「一筆訂單成立之後，到底會發生幾件事？」沒有人答得出來，因為答案散在十二個訂閱者的程式碼裡。

事件驅動架構（Event-Driven Architecture, EDA）是解耦手段裡最強力的一種：它不只解掉「誰依賴誰」，連「誰等誰」都解掉了。這篇筆記講清楚它買到什麼、以及那句經典提問代表的帳單。

## 技術背景

### 先破除誤解：用了訊息佇列不等於事件驅動

判斷標準不在基礎設施，在**訊息的語意**：

| | 命令（Command） | 事件（Event） |
|---|---|---|
| 語意 | 「你去做這件事」 | 「這件事已經發生了」 |
| 指向 | 指定的接收者 | 不指定，誰關心誰訂閱 |
| 耦合 | 發送方知道接收方的職責 | 發送方不知道有誰在聽 |
| 拒絕的權利 | 接收方執行失敗，發送方要處理 | 訂閱者怎麼反應是它家的事 |

透過佇列發送「幫我發簡訊」給通知服務——那是**非同步化的 RPC**，呼叫方依然知道對方是誰、期待對方做什麼，耦合只是從時間軸上被藏起來。真正的事件是 `OrderCreated`：一個過去式的事實宣告，發布者的責任到此為止。命令沒有錯，錯的是把「上了 MQ」當成「解耦了」。

### 兩種拓撲：編排與協奏

多步驟流程（下單 → 扣庫存 → 發票 → 通知）在事件驅動下有兩種組織方式：

- **協奏（choreography）**：沒有指揮。每個服務聽上一步的事件、做自己的事、再發出自己的事件，流程是**湧現**出來的。加一步不用改任何人；但流程全貌不存在於任何一份程式碼裡。
- **編排（orchestration）**：有一個編排者依序發令並處理結果。流程集中可見、失敗補償好寫；代價是編排者成為耦合點與單點。

經驗法則：**步驟少、各自獨立的廣播型後續動作用協奏；有嚴格順序、需要補償回滾的關鍵流程用編排**。兩者在同一系統中並存是常態而非妥協。

### 事件該帶多少資料

Martin Fowler 在〈What do you mean by "Event-Driven"?〉（2017）中區分的前兩種模式，是每個事件設計都要選邊的題目：

- **事件通知（Event Notification）**：薄事件，只帶 ID——「訂單 123 成立了」。訂閱者要細節就回頭查。耦合小、事件穩定，但producer 被查詢流量反打，且「回查時狀態已變」是經典陷阱。
- **事件攜帶狀態轉移（Event-Carried State Transfer）**：胖事件，帶上完整快照。訂閱者自給自足、不回查；代價是各處留有資料複本，schema 一改天下動。

### 上車前的必答題

事件驅動把同步呼叫的顯性失敗，換成了一組必須主動回答的問題——**每一題都沒有預設答案**：

1. **重複**：訊息中介多半保證「至少一次」，訂閱者必然收到重複事件——冪等處理不是加分題（見規劃中的〈冪等性：重試安全的前提〉）。
2. **丟失**：「寫庫成功、發事件失敗」的半途狀態怎麼辦——見規劃中的〈分散式交易：2PC、Saga 與 Outbox〉。
3. **順序**：`OrderCancelled` 比 `OrderCreated` 先到，訂閱者會怎樣？
4. **時差**：最終一致的「最終」是多久？業務方接受嗎——這是品質屬性問題，回到〈[品質屬性與取捨](../01-architecture-thinking/quality-attributes-and-tradeoffs.md)〉的追問法。

## 實際案例

### 上半場：事件驅動兌現承諾

訂單完成後要做五件事：發點數、開發票、通知物流、更新報表、風控留痕。同步鏈的寫法，五個依賴全掛在下單流程上——發票服務一慢，全體下單變慢；風控服務一掛，訂單建不成。

改成發布 `OrderCompleted` 事件後：下單流程只剩「存庫＋發事件」，回應時間與五個下游徹底無關；發票服務維護中，事件堆在佇列裡等它回來慢慢消化；三個月後新增「完成訂單送抽獎券」，抽獎服務自己訂閱就好，**訂單服務一行未改**。解耦、削峰、失敗隔離，全部兌現。

### 下半場：帳單寄到

半年後的三個現場：

- **查案變成跨服務考古**。客訴「點數沒發」，值班工程師得先知道「發點數是訂閱 `OrderCompleted` 的十二個服務之一」，再去點數服務翻消費紀錄、比對佇列位移——因果鏈在系統圖上是看不見的，只存在於佇列與日誌裡。沒有分散式追蹤與統一的事件目錄，每次查案都是考古。
- **胖事件的 schema 變成公共財**。`OrderCompleted` 帶了完整訂單快照，某訂閱者悄悄依賴了其中的 `promotionTag` 欄位。訂單團隊某次「內部重構」移除該欄位——他們根本不知道有人在用——下游在生產環境炸開。發布者以為自己沒有消費者，正是事件驅動的耦合幻覺：**依賴沒有消失，只是從編譯期躲進了執行期**。
- **重複事件照進現實**。佇列重投遞造成同一筆 `OrderCompleted` 被消費兩次，點數發了兩份——冪等這道必答題，當初被當成了選答。

這些不是失敗案例，是**價目表**。這套系統的下單可用性確實不再受十二個下游牽制；代價是可觀測性、schema 治理、冪等設計從「進階題」變成「地基」。

## 技術優缺點

**優勢**

- **發布者與訂閱者互不相識**：新增消費者零成本，系統以「加訂閱」的方式演化。
- **時間解耦**：下游慢或掛，上游無感；佇列天然削峰。
- **失敗隔離**：一個訂閱者的故障不會外溢到發布者與其他訂閱者。

**代價**

- **因果鏈不可見**：流程不存在於任何單一程式碼中，理解與除錯依賴追蹤基礎設施。
- **執行期耦合**：事件 schema 是沒有編譯器保護的公共 API，治理（版本、目錄、消費者登記）必須人工建立。
- **最終一致的溝通成本**：業務與客服必須理解並接受「剛下單，點數幾秒後才到」。
- **必答題清單**：冪等、丟失、順序——每個訂閱者都要各自答對一遍。

**這改變了什麼實務決策**：事件適合「發生了一件事，後續動作各自獨立」的廣播型場景；強順序、需同步確認、失敗要立即回滾的核心步驟，同步呼叫或編排反而誠實。以及一條硬規則：**可觀測性與事件治理要和第一個事件一起上線**，而不是等第一次跨服務查案之後。

## 小結

- 事件驅動的判準是語意不是設施：事件是「已發生的事實」，不指定誰做什麼；用佇列發命令只是非同步 RPC。
- 協奏讓流程湧現、加步驟免改人；編排讓流程可見、補償好寫——關鍵流程用編排，廣播場景用協奏。
- 薄事件回查、胖事件複本，各有各的耦合；schema 是沒有編譯器保護的公共 API。
- 解耦不消滅依賴，只是把它從編譯期移到執行期——換來的自由，用冪等、治理與可觀測性支付。
- 事件驅動常被當成微服務的標配，但「要不要拆成那麼多服務」本身才是更早該問的問題——本章最後一篇：〈[單體 vs 微服務：什麼時候該拆](monolith-vs-microservices.md)〉。

## 常見面試題

1. **事件（event）和命令（command）差在哪？為什麼用了 MQ 不一定是事件驅動？**（提示：語意與耦合方向；「非同步 RPC」這個詞說出來就到位）
2. **choreography 和 orchestration 怎麼選？**（提示：流程可見性與補償需求 vs 演化自由度；答「並存」不算和稀泥）
3. **事件驅動系統上線前，你會檢查哪些「必答題」？**（提示：重複／冪等、丟失／Outbox、順序、最終一致的業務溝通，外加可觀測性）
