# 快取策略：Cache-Aside、雪崩、穿透與擊穿

## 前言

效能會議的萬用結論：「加個快取就好了。」三個月後的值班群組：「快取到期的那一秒，資料庫連線池瞬間打滿，全站超時。」——快取不是加速器這麼單純的東西；它是一台**取捨機器**：收進去的是延遲與資料庫負載，吐出來的是一致性降檔和三種全新的故障模式。

用上一章的語言說：加快取就是在 PACELC 的 E 側選了 L（〈[CAP 與 PACELC](../04-distributed-basics/cap-and-pacelc.md)〉）——你主動為這條讀路徑引入了一份**會過期的複本**。這篇筆記講清楚讀寫模式怎麼選、三災（穿透、擊穿、雪崩）怎麼防，以及那個最常被當成技術參數的東西——TTL——為什麼其實是業務決策。

## 技術背景

### 先破除誤解：快取不是無副作用的加速器

快取是一份**你承諾它會過期的複本**。從掛上快取那一刻起，這條讀路徑就從「讀資料庫的真相」變成「讀一份最多舊 TTL 秒的快照」——你引入了最終一致（〈[一致性模型](../04-distributed-basics/consistency-models.md)〉整篇適用）。「加快取」的完整句子是：「**這個資料允許舊 N 秒，換取延遲與資料庫負載的下降**」——N 填不出來，表示這個快取還沒被想清楚。

### 讀寫模式：Cache-Aside 與它的兄弟們

| 模式 | 讀 | 寫 | 適用 |
|---|---|---|---|
| Cache-Aside（旁路） | 先查快取，miss 則查庫並回填 | 寫庫，然後**刪**快取 | 預設選擇，應用全權掌控 |
| Read-Through | 快取層自己去庫裡載 | — | 快取元件支援時，簡化應用碼 |
| Write-Through | — | 寫快取，快取同步寫庫 | 讀多寫少、要求寫後立即可讀 |
| Write-Behind | — | 寫快取，非同步批次刷庫 | 寫入極熱；**代價是掉電丟資料** |

Cache-Aside 的寫路徑有一個常年被問的選擇題：**為什麼是「刪快取」不是「更新快取」？**因為兩個並發寫入的「更新快取」可能亂序到達——寫庫順序是 A→B，更新快取順序卻是 B→A，快取從此留著舊值直到 TTL 救場。「刪除」讓下一個讀者回源重建，把「以誰為準」收斂回資料庫這唯一真相。即便如此，「先寫庫、再刪快取」仍有一道微小的殘留窗口（刪除後、回填前，另一個讀者帶著更舊的值回填）——工程上以「TTL 兜底＋窗口機率極低」接受它；追求更嚴的一致性要付出顯著複雜度，多數場景帳算不平。

### 三災：穿透、擊穿、雪崩

| 災 | 成因 | 藥 |
|---|---|---|
| 穿透（penetration） | 查詢**不存在**的 key，快取永遠 miss，每發都打庫——常見於惡意掃描或髒資料 | 空值也快取（短 TTL）；入口擋掉非法 key 格式；布隆過濾器擋「一定不存在」 |
| 擊穿（breakdown） | 單一**熱 key** 過期瞬間，成千上百個並發讀同時回源重建 | 互斥重建（只放一個請求回源，其餘等）；邏輯過期（物理不過期、背景刷新） |
| 雪崩（avalanche） | **大量 key 同時過期**，或快取整體故障，讀流量全額砸向資料庫 | TTL 加隨機抖動；預熱錯峰；快取故障時的降級開關與限流兜底 |

三災共享同一個本質：**快取平時擋掉了九成讀流量，資料庫的容量就是按剩下一成配的**——任何讓流量繞過快取的瞬間，打到的都是一個承受不了全額流量的後端。快取越有效，後面的系統越脆弱，這是它最反直覺的性質。

## 實際案例

大促前的商品頁快取，一場雪崩的完整推演：

**準備**：運維在 23:00 跑預熱腳本，把三萬個商品頁塞進快取，TTL 統一設 3600 秒。當下一切健康：快取命中率 98%，資料庫只吃 2% 的讀流量，連線池配置也是按這個水位給的。

**午夜 00:00**：三萬個 key 在同一秒到期。此刻正是活動開場、流量最高的一刻——所有商品頁請求同時 miss、同時回源。資料庫瞬間承受 50 倍於平時的查詢，連線池佔滿；接著是連鎖反應：等不到連線的請求堆積在應用執行緒池 → 應用全體變慢 → 健康檢查超時 → 節點被摘除 → 剩餘節點壓力更大。**快取一切正常，只是三萬個承諾在同一秒同時到期。**

**三帖藥**，對應三個層次：

1. **TTL 抖動**（防同時到期）：`3600 + random(0, 600)` 秒——預熱腳本一行之改，到期曲線從懸崖變緩坡。
2. **熱 key 互斥重建**（防單點擊穿）：頭部商品即使錯峰，單一 key 到期仍有百併發回源——重建加互斥，一人回源、眾人稍候或暫拿舊值。
3. **降級開關**（防快取整體故障）：快取叢集真的掛掉時，商品頁退回精簡版靜態內容＋資料庫前置限流——寧可頁面樸素，不可全站陪葬。這一帖平時不跑，屬於〈品質屬性〉裡「要贏什麼、不能輸什麼」的後半句。

## 技術優缺點

**優勢**

- **延遲與成本**：熱資料的讀取從資料庫量級（毫秒級＋連線成本）降到記憶體量級。
- **保護後端**：擋掉的讀流量直接轉化為資料庫容量餘裕——多數系統的讀寫比讓這筆帳極其划算。
- **天然的降級素材**：後端故障時，「舊資料」常常比「沒資料」有用（上一章買不到的可用性，這裡用舊度換到一點）。

**代價**

- **一致性降檔**：讀到的是快照；「舊 N 秒」必須被業務明文接受。
- **新故障模式**：三災是快取**特有**的病——沒有快取的系統不會雪崩。
- **失效邏輯的複雜度**：刪除時機、雙寫窗口、預熱與淘汰策略，每一項都是新的出錯面。
- **脆弱性轉移**：命中率越高，後端容量越按「剩餘流量」配置，繞過快取的瞬間越致命。

**這改變了什麼實務決策**：加快取的評審問題不是「要不要加」，而是三連問：「**這資料允許舊多久（TTL 是業務答案）？這 key 的到期瞬間誰扛（擊穿方案）？快取全掛時走哪條路（降級方案）？**」三題答齊，快取才算設計完成——只答了第一題的快取，是把後兩題留給值班的人。

## 小結

- 快取是「承諾會過期的複本」：加快取＝為這條路徑選擇最終一致，「允許舊多久」是業務決策不是技術參數。
- Cache-Aside 寫路徑用「刪」不用「更」：讓真相收斂回資料庫；殘留窗口以 TTL 兜底、明知而接受。
- 三災一本質：穿透（不存在的 key）、擊穿（熱 key 到期）、雪崩（集體到期／整體故障）——都是流量繞過快取砸向按殘量配置的後端。
- 藥方成套：空值快取＋布隆、互斥重建＋邏輯過期、TTL 抖動＋降級開關——三題答齊才算設計完成。
- 快取越有效，後端越脆弱——命中率是資產也是負債。
- 快取吸收的是**讀**的洪峰；寫的洪峰呢？把時間軸當緩衝的另一件工具：〈[訊息佇列模式：削峰、解耦與順序性](message-queue-patterns.md)〉。

## 常見面試題

1. **Cache-Aside 為什麼建議刪快取而不是更新快取？**（提示：並發寫亂序回填；「讓真相收斂回資料庫」一句到位）
2. **穿透、擊穿、雪崩怎麼區分、各怎麼防？**（提示：不存在的 key／熱 key 到期／集體到期；答得出「共同本質是流量繞過快取」是加分）
3. **快取和資料庫的一致性能做到什麼程度？**（提示：Cache-Aside 是最終一致＋TTL 上界；說得出「為什麼不追求強一致」比背出方案更對）
