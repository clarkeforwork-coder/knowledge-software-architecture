# Back-pressure：當下游跟不上

## 前言

「消費者追不上，佇列一直長。加了機器還是追不上。緩衝區再開大一點？」——這條路的終點是固定的：記憶體耗盡、行程重啟、緩衝區裡的資料全數蒸發。**為了不丟資料而放大的緩衝，最後以丟掉所有資料收場**——這是分散式系統最諷刺的死法之一。

前一篇的限流是「丟」、熔斷是「止損」，都發生在邊界上的一刀。這一篇談第三種、也是最被低估的一種姿勢：**回壓（back-pressure）——把「我跟不上了」這個訊號沿著鏈路往上游傳，讓生產端自己慢下來**。它不丟資料、不斬呼叫，代價是全鏈路都得聽得懂這個訊號。

## 技術背景

### 先破除誤解：緩衝區不會吸收一切

「加大佇列」對付得了**尖峰**，對付不了**持續超載**——這條界線由一條 1961 年就被證明的定律劃定。Little's Law：

> L = λ × W（隊伍長度 ＝ 到達率 × 平均等待時間）

當到達率 λ 持續大於服務率，隊伍長度 L 沒有穩態——它線性增長，直到撞上某個物理極限；而等待時間 W 跟著同步惡化。換句話說：**無界緩衝不能消化超載，只能把「現在就失敗」換成「更晚、更大、帶著延遲爆炸的失敗」**。〈[訊息佇列模式](message-queue-patterns.md)〉裡「削峰的前提是峰後有谷」，就是這條定律的口語版。

所以第一原則：**所有緩衝都應該有界**。有界不是限制，是誠實——它逼系統在緩衝滿的時候回答那個遲早要答的問題：接下來怎麼辦？

### 緩衝滿了的三種回答

1. **丟（load shedding）**：新來的不收，或按優先級丟——上一篇限流的親戚。適合可丟的資料：指標取樣、日誌、即時性強於完整性的訊號。
2. **繼續緩衝**：只在確知是短尖峰時合法；對持續超載就是上面那條諷刺的死路。
3. **回壓**：緩衝滿 → 對生產端的寫入變慢或阻塞 → 生產端感知到「下游滿了」→ 生產端自己減速，或把訊號**再往上傳**。流量控制訊號逆著資料流的方向傳播，直到抵達一個能真正做決定的地方。

### 回壓的機制長什麼樣

它不是新發明——你每天都在用：

- **TCP 滑動視窗**：接收方通告「我還能收多少」，發送方超過就得等——網際網路本身就靠回壓活著，這是最古老也最成功的實作。
- **有界佇列＋阻塞寫入**：行程內最樸素的形式——佇列滿，`put()` 就是不返回，生產執行緒被迫等待。同步呼叫鏈其實**天然自帶回壓**（呼叫方等回應期間無法製造新請求）；正是「改成非同步」切斷了這條天然通道，才需要顯式把它接回來。
- **Pull 模型**：消費者主動拉、拉多少自己說——「我來拿」天然就是回壓，速率的決定權在慢的那一方。Reactive Streams 規範的 `request(n)` 把這個語意標準化：下游先聲明需求量，上游至多發這麼多。
- **Credit-based 流控**：上游持有下游發的「額度」，額度用完即停，下游消化後再補發——TCP 視窗與 `request(n)` 的一般化形式。

### 回壓的阿基里斯腱：一個斷點就全盤失效

回壓是**鏈**：A → B → C，C 慢了，訊號要經 B 傳回 A。鏈路上任何一個**無界緩衝**都會吃掉訊號——B 對 C 的慢無動於衷、照單全收堆在自己肚子裡，A 就永遠感知不到，直到 B 爆炸。**回壓的有效性取決於鏈上最寬的那個緩衝**，這讓它成為端到端的設計，而不是單點的配置。

鏈的最上游是誰？往往是**使用者**。回壓傳到底的形態是誠實的產品體驗：按鈕轉圈、排隊頁、「系統忙碌請稍候」——比起假裝收下然後靜默丟失，讓上游（哪怕是人）知道「現在慢」永遠是更誠實的合約。

## 實際案例

一條事件管線：應用程式 → 收集代理（agent）→ 中心儲存。某日儲存端做索引重建，寫入吞吐掉一半。兩個版本的推演：

**無界版**。agent 的發送緩衝是無界的記憶體佇列——設計初衷是「別因為儲存抖動就丟事件」。儲存變慢後：agent 記憶體 30 分鐘內從 200MB 爬到 4GB → OOM 被系統擊殺 → 重啟 → **緩衝裡 4GB 的事件無聲蒸發** → 更糟的是重啟後的 agent 立刻被應用端積累的補發流量再次灌滿，進入 OOM 循環。想保所有資料的設計，丟掉了最多資料，還把「儲存變慢」升級成「採集層反覆崩潰」。

**有界＋回壓版**。agent 緩衝設上限並分級：緩衝到 70%，對**可丟級**（除錯日誌、全量指標）啟動降採樣——這是局部的 shed；到 90%，對**不可丟級**（審計事件、計費記錄）的寫入 API 開始阻塞——回壓傳進應用程式，應用的事件寫入呼叫變慢，關鍵路徑的 P99 從 80ms 升到 140ms，**慢，但一筆未丟**。監控同時亮起兩盞燈：agent 緩衝水位、應用寫入延遲——值班的人在系統還活著的時候就知道儲存病了，而不是在 OOM 之後考古。

儲存修復後，緩衝水位回落、採樣恢復、延遲歸位——全程沒有資料丟失，沒有行程死亡。兩版的差別不是誰的程式碼多，是**設計時有沒有回答「滿了怎麼辦」**。

## 技術優缺點

**優勢**

- **過載下優雅降速而非崩潰**：系統的失效模式從「爆炸」變成「變慢」，而慢是可觀測、可反應的。
- **資料不無聲丟失**：丟是顯式決策（分級 shed），不是 OOM 的副作用。
- **延遲可預期**：有界緩衝＋Little's Law 給出等待時間的上界，SLO 才有計算基礎。

**代價**

- **端到端的實作成本**：每一跳都要有界、都要傳訊號——一個無界斷點讓全鏈白做；非同步鏈路要顯式設計（同步鏈免費的東西，非同步要自己買回來）。
- **上游體驗變差要業務簽字**：回壓的終點是「請求變慢」或「使用者排隊」——這是產品決策，不是純技術參數。
- **與吞吐的張力**：太小的有界緩衝把正常抖動也放大成回壓波動；緩衝尺寸與水位線是需要觀測調校的參數。

**這改變了什麼實務決策**：架構圖上每一條非同步的邊，評審時多三個必答格：「**緩衝有界嗎？滿了丟誰？訊號傳給誰？**」三格填齊，這條邊才算設計完成。以及一條選型原則：可丟的資料用 shed 保系統，不可丟的資料用回壓保資料——**先分級，再選姿勢**，兩者在同一條鏈上可以並存（案例裡的 70%／90% 雙水位正是如此）。

## 小結

- Little's Law 是鐵律：持續超載下無界緩衝沒有穩態——它把小失敗攢成大失敗，最後 OOM 一次付清。
- 所有緩衝都該有界；有界逼你回答「滿了怎麼辦」：丟（shed）、短暫緩衝、或回壓。
- 回壓＝流控訊號逆流：TCP 視窗、阻塞寫入、pull／`request(n)`、credit——同步鏈天然自帶，非同步鏈要顯式接回。
- 回壓是端到端的鏈：一個無界斷點就吃掉訊號；傳到最上游的形態常常是誠實的使用者體驗。
- 先給資料分級，再選姿勢：可丟用 shed 保系統、不可丟用回壓保資料，雙水位並存。
- 本章四篇——快取、佇列、閘門、回壓——每一篇的結論都是一組取捨與參數。三年後還有人記得「當初為什麼這樣設」嗎？最後一章就管這件事：從〈規劃中的「ADR：把『為什麼』留下來」〉開始。

## 常見面試題

1. **為什麼說無界佇列危險？**（提示：Little's Law——持續超載下 L 與 W 無穩態；「想保資料反而 OOM 全丟」的悖論）
2. **回壓訊號怎麼從最下游傳到最上游？舉兩種機制。**（提示：TCP 視窗、有界佇列阻塞、pull／request(n)、credit；關鍵詞「一個無界斷點就失效」）
3. **丟（shedding）、緩衝、回壓怎麼選？**（提示：先按可丟性分級；短尖峰才配緩衝；答得出「同鏈可並存、分水位觸發」是加分）
