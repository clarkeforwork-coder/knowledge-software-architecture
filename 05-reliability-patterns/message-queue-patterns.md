# 訊息佇列模式：削峰、解耦與順序性

## 前言

「加個 MQ 解耦一下。」——訊息佇列進場的理由，常常就這麼一句。但 MQ 其實同時在賣三種商品：**削峰**（把尖峰攤到時間軸上）、**解耦**（收發雙方互不等待、互不相識）、**非同步**（呼叫端先走）；而不管你買哪一種，付的都是同一組代價：延遲加一跳、順序與重複成為日常、積壓成為新的監控對象。

〈[事件驅動架構](../02-architecture-styles/event-driven-architecture.md)〉談的是**語意**——事件與命令、協奏與編排；這一篇下沉一層，談**佇列本身的力學**：削峰的前提、順序的價格、毒訊息的下場。語意選對了，力學不懂，佇列照樣把你絆倒。

## 技術背景

### 先破除誤解：MQ 不會讓系統變快

單看一次請求，MQ 讓它**變慢**了——多一跳網路、多一次落盤。MQ 買的從來不是低延遲，是**吞吐的平滑**與**失敗的隔離**：下游每秒只能處理一千件，尖峰每秒來五千件，佇列把多出來的四千件存進時間軸，讓下游按自己的節奏消化。

但削峰有一個常被略過的前提：**峰是暫時的**。生產速率短暫超過消費速率，佇列是緩衝；生產速率**持續**超過消費速率，佇列是一條越來越長的死緩——積壓無限增長，端到端延遲跟著無限增長（這條路的盡頭在〈[Back-pressure](backpressure.md)〉再談）。削峰填谷的完整句子是：「峰之後必須有谷，讓消費端還債。」

### 兩種消費模型

- **競爭消費者（佇列語意）**：一則訊息只被消費群組裡的**一個**成員處理——這是水平擴展消費能力的機制：消費者不夠就加人，佇列自動分工。
- **發布訂閱（主題語意）**：一則訊息投遞給**每一個**訂閱群組——事件驅動的「誰關心誰訂閱」靠它實現。

兩者常常疊用：主題對多個消費**群組**廣播，每個群組內部競爭消費。搞混兩層的結果通常是「訊息被搶走了」或「訊息被處理了兩遍」的假 bug。

### 順序性：全域順序是奢侈品

「訊息會按發送順序到達嗎？」——預設答案是**不會**，而且多數時候你也買不起「會」：全域順序意味著全域只有一條處理線，吞吐與可用性同時歸一。工程上的甜蜜點是**分區內順序**：

- 佇列分成 N 個分區，**單一分區內**保證先進先出；
- 生產端按業務鍵路由——同一張訂單的事件永遠進同一個分區，於是「同一張訂單的事件有序」，不同訂單之間無序（而它們本來也不需要互相有序）。

代價立刻浮現：**分區數是並行度的上限**（一個分區同一時間只有一個消費者在吃），而按 key 路由引入**熱鍵傾斜**——大客戶的訊息全擠在一個分區，別的分區閒著，它的隊伍排到天邊。順序、並行、均衡三者互相拉扯，分區鍵的選擇是這道題的答案欄。

### 失敗的力學：ack、重投與毒訊息

- **確認時機**：處理完才 ack，中途當機訊息會重投——at-least-once 的來源，也是〈[冪等性](../04-distributed-basics/idempotency.md)〉是消費端義務的原因。先 ack 再處理則是 at-most-once：當機即丟，只適合可丟資料。
- **毒訊息（poison message）**：一則永遠處理失敗的訊息（格式錯、反序列化炸、觸發程式 bug）。不處理它的下場見下方案例——它會卡死整個分區。
- **死信佇列（DLQ）**：重試 N 次仍失敗的訊息移到旁路佇列，主隊伍繼續前進。兩條紀律：**DLQ 必須有告警與處理流程**——沒人看的 DLQ 只是資料墳場；入 DLQ 的訊息要帶齊上下文（原始內容、失敗原因、重試次數），否則事後考古無從下手。
- **積壓（lag）監控**：佇列系統的第一健康指標不是吞吐，是**消費落後量**。lag 短暫上升＝削峰在工作；lag 持續上升＝消費端病了或峰不是峰——兩種讀法對應兩種完全不同的處置。

## 實際案例

大促下單鏈路的佇列設計，一次走齊四個決策點，再看一場毒訊息事故：

**設計**：下單 API 完成本地交易後（經 Outbox）發出事件；後續的積分、發票、物流各為一個消費群組訂閱同一主題（發布訂閱層）；每個群組內部多個實例競爭消費（佇列層）。分區鍵選 `orderId`——同一張訂單的「成立→付款→出貨」有序，不同訂單並行。發票服務處理較慢，單獨給它更多消費實例——**各群組獨立伸縮**，正是解耦紅利的具體形狀。

**事故推演**：某次上游改版，一則訂單事件帶了非法的日期格式。發票服務反序列化拋例外——沒有 DLQ 的配置下，訊息不被 ack、佇列忠實重投、再炸、再重投……這個分區的隊伍從此停在這一則訊息後面（**隊頭阻塞**）：這個分區裡所有訂單的發票全部延遲，而監控面板上服務「活著」、CPU 很低、只有 lag 在一路爬升。修復要兩層：當下把毒訊息手動移走讓隊伍前進；長期配置「重試 3 次＋退避 → 入 DLQ ＋告警」，讓一則壞訊息的爆炸半徑收斂為一張工單，而不是一個分區的癱瘓。

## 技術優缺點

**優勢**

- **削峰**：尖峰被時間軸吸收，下游按自己的容量消化——前提是峰後有谷。
- **失敗隔離與獨立伸縮**：消費群組各自故障、各自擴縮，互不牽連。
- **重放能力**：訊息落盤的佇列可以回撥位移重新消費——事故修復後「補跑」的底氣。

**代價**

- **延遲加一跳**：同步鏈路變非同步，端到端延遲的下界抬高，且尾部受積壓影響。
- **三題常駐**：順序（分區與熱鍵）、重複（冪等）、積壓（lag 監控與還債容量）——上了佇列就是終身題。
- **在途資料的不可見**：資料「在佇列裡」的狀態對業務查詢是黑箱，「我的發票呢」需要新的可觀測手段。
- **多一套要運維的基礎設施**：佇列自身的可用性、容量、升級，全是新帳。

**這改變了什麼實務決策**：引入佇列的第一個問題不是選型，是**業務題**：「這條路徑能不能非同步——呼叫端不等結果，業務還成立嗎？」成立，才有資格談削峰與解耦；不成立（下單必須當場知道成敗），硬塞佇列只是把同步等待改寫成輪詢等待。第二個問題才是工程題：分區鍵選誰、重試幾次入 DLQ、lag 告警線畫在哪。

## 小結

- MQ 賣三種商品（削峰、解耦、非同步），收一組固定代價（延遲＋順序＋重複＋積壓）；它不讓系統變快，讓系統變平滑。
- 削峰的前提是峰後有谷：持續的生產大於消費，佇列只是死緩——那是回壓的主場。
- 順序的甜蜜點是分區內有序＋按業務鍵路由；分區數是並行上限，熱鍵是新的傾斜來源。
- 毒訊息不處理就是隊頭阻塞：重試有上限、DLQ 有告警、入隊帶上下文——三條紀律缺一不可。
- lag 是佇列系統第一指標：短升是削峰在工作，長升是系統在說謊。
- 佇列保護的是**非同步**路徑；同步路徑的過載——流量直接打在 API 上、慢下游拖住你的執行緒——需要另外兩道閘門：〈[限流與熔斷：保護系統的兩道閘門](rate-limiting-and-circuit-breaker.md)〉。

## 常見面試題

1. **MQ 削峰的前提是什麼？什麼情況下削峰失效？**（提示：峰後有谷；持續性超載下積壓無限增長——說得出 lag 兩種讀法更好）
2. **怎麼保證訊息順序？代價是什麼？**（提示：分區內順序＋業務鍵路由；分區數＝並行上限、熱鍵傾斜）
3. **毒訊息會造成什麼？怎麼設計防護？**（提示：隊頭阻塞卡死分區；重試上限＋退避＋DLQ＋告警）
