# 事故追讀：GitHub 2018-10-21——43 秒分區，24 小時債

## 前言

〈[CAP 與 PACELC](cap-and-pacelc.md)〉說「分區是天氣不是選項」，聽起來像修辭。這篇深入文用一份一手資料證明它是實況：2018 年 10 月 21 日，GitHub 因一次例行的光纖設備更換，美東網路樞紐與主資料中心斷連——**43 秒**。連線恢復時，系統已經走進一個無法安全回頭的狀態，最終以 **24 小時 11 分**的降級服務收場。

追讀的材料是 GitHub 於 2018-10-30 發布的官方事後分析（〈October 21 post-incident analysis〉，見延伸閱讀）；本文所有數字與引述皆出自該文。值得追讀的原因很純粹：這場事故裡**沒有人犯錯、沒有程式 bug**——每個元件都按設計正確運作，然後整個系統掉進了第四章描述的每一個坑。教科書案例通常是簡化的；這一份是真的。

## 技術背景

### 先破除誤解：「有自動 failover，分區就不可怕」

這場事故最反直覺的地方：主角不是故障的設備，而是**正確運作的高可用工具**。GitHub 當時的 MySQL 拓撲由 Orchestrator 管理——一套基於 Raft 共識的 failover 協調系統，設計目標正是「主庫失聯時自動提升新主庫」。分區發生的 43 秒內，它做了它該做的每一件事。

事故因此值得每個「裝了 HA 就安心」的團隊細讀：**自動 failover 回答的是「主庫死了怎麼辦」，而分區製造的問題是「主庫沒死，只是你看不見它」**——這兩題的正確答案不同，但自動化只會答第一題。

### 追讀前需要的拓撲背景

依官方分析與 GitHub 同年發布的架構文章：美東實體資料中心為主站（MySQL 主庫所在），美西資料中心與多個公有雲節點作為複本與 Orchestrator 的共識參與者；跨岸複寫為非同步——這一點是整場事故的伏筆：**非同步複寫意味著任一瞬間，主庫上永遠有一小段「已提交、未複寫」的寫入**（PACELC 的 E 側選了 L，代價以毫秒計——直到分區來收帳）。

## 實際案例

### 第一幕：43 秒（22:52 UTC）

例行維護更換故障的 100G 光學設備，造成美東網路樞紐與美東主資料中心之間斷連。報告原文：「Connectivity between these locations was restored in 43 seconds.」——放在多數系統的監控圖上，這只是一根毛刺。

### 第二幕：自動化的正確反應

43 秒內，美東的 Orchestrator 節點失去 quorum、開始卸任領導；美西與公有雲的 Orchestrator 節點**達成 quorum**，依設計把資料庫叢集的主庫提升到美西。注意：從美西節點的視角，「美東主庫失聯」與「美東主庫死亡」**在資訊上不可區分**——這正是分區的定義。自動化在它的世界觀裡做了唯一正確的事。

### 第三幕：分歧既成

連線恢復後，世界變成這樣：美東的原主庫上有一段**已提交但未複寫到美西**的寫入——最繁忙的叢集在那個窗口有 **954 筆**；而美西的新主庫已經開始接受新寫入。報告的關鍵句：

> 「因為兩個資料中心的資料庫叢集現在各自持有對方沒有的寫入，我們無法安全地把主庫切回美東。」

用第四章的語言：這是**沒有無害合併函數的分歧**——購物車可以取聯集，Git repo 的元資料、issue 編號、權限變更不行。43 秒的分區把系統送進 CAP 的考場，自動化在鈴響前就交了卷，而卷子上的選擇（可用性優先、跨區提升）從此不可撤銷。

### 第四幕：抉擇——顯式的 C over A

GitHub 的決定，報告寫得毫不含糊：「**prioritize data integrity over site usability**」（資料整全性優先於網站可用性）。具體形狀：暫停 webhook 投遞與 GitHub Pages 建置——這兩個系統會對外產生不可回收的副作用，在資料狀態未收斂前寧可停；網站本體維持服務但部分資料過時，並持續公告。注意粒度：**不是全站下線，而是逐服務選邊**——第四章「逐資料、逐操作選 C 或 A」的原則，在事故中的形狀就是這樣。

### 第五幕：還債 24 小時

修復路徑暴露了帳單的每一行：備份每 4 小時一輪、多 TB 的資料要還原（後改用美西直傳美東加速）、複寫追趕耗時——合計 **24 小時 11 分**的降級。期間積壓了**超過 500 萬則 webhook 事件與 8 萬個 Pages 建置**；恢復後重放這些事件——**約 20 萬則超過內部 TTL 的 webhook payload 被直接丟棄**（一個顯式的 shed 決策），其餘全量補發。對所有 webhook 消費端來說，這是一場真實世界的 at-least-once 大考：晚到 24 小時的事件重放，冪等沒做好的下游在這一天現形。

### 追讀的收穫：他們修了什麼

官方列出的改善項，每一條都對著本章的一個概念：**限制 Orchestrator 不得跨區提升主庫**（給自動化的權限加上人類確認的閘——分區時寧可短暫不可用，不再自動走進不可回頭的分歧）；狀態回報從紅黃綠改為**逐元件粒度**；加速 **active/active/active** 三活架構（讓「失去一個站點」不再等於「失去寫入能力」）；以及**常態化的故障演練**——這場事故的恢復之所以要 24 小時，一半原因是這條恢復路徑從未被完整演練過。

## 技術優缺點

把這場事故當一個「設計」來評析：

**GitHub 做對的**

- **整全性優先的膽識**：頂著 24 小時的商譽損失守住資料正確——事後沒有任何使用者資料錯亂的報告，這是那 954 筆寫入之外的全部賭注。
- **誠實的降級**：暫停會產生外部副作用的服務（webhook、Pages），而不是讓它們基於分歧的資料繼續對外開火。
- **一手報告本身**：時間線、數字、根因、改善項全公開——這份文件的存在就是〈[ADR](../06-decisions-and-documentation/adr.md)〉精神的事故版。

**帳單上的教訓**

- **自動化的權限超過了它的判斷力**：Orchestrator 有能力跨區提升，卻沒有能力判斷「這 43 秒是分區還是死亡」——**能力與權限的落差，就是事故的形狀**。修復項第一條正是收權。
- **E 側的帳單平時看不見**：非同步複寫每天省下毫秒級延遲，直到分區那天，954 筆未複寫寫入變成 24 小時的錨。
- **未演練的恢復路徑不算存在**：備份存在、可還原，但「多 TB 跨國還原要多久」是事故當天才第一次量測的數字。

**這改變了什麼實務決策**：檢查你自己系統裡每一個「會自動做不可逆決策」的元件（failover 工具、自動擴縮、自動清理），問同一題——**它分得清「失聯」與「死亡」嗎？分不清的話，它的動作半徑該不該跨過那條不可回頭的線？**

## 小結

- 43 秒分區 → 自動跨區提升 → 兩岸各持對方沒有的寫入 → 不可安全回切 → 24 小時 11 分還債：每一步都「正確」，整體卻是事故——分散式失效是湧現的，不是某個元件的錯。
- 954 筆未複寫寫入是 PACELC E 側的具象帳單：非同步複寫平時省延遲，分區時付本金。
- 「integrity over usability」是 CAP 抉擇的實名版本，而且是逐服務的粒度：停 webhook/Pages、網站帶舊資料續命。
- 自動化只會答「死了怎麼辦」，答不了「只是看不見怎麼辦」；能力與權限的落差就是事故的形狀。
- 恢復路徑要演練，webhook 消費端的冪等在 500 萬則重放面前無所遁形——第四章每一篇，都在這一天被點名。
- 讀完本文，建議把官方報告原文與〈[CAP 與 PACELC](cap-and-pacelc.md)〉〈[一致性模型](consistency-models.md)〉〈[冪等性](idempotency.md)〉對讀一輪——概念與實況互相印證，是 🔬 軌存在的理由。

## 常見面試題

1. **用一個真實案例說明 CAP 的取捨。**（提示：這一篇就是答案模板——43 秒分區、跨區提升、整全性優先；說得出「逐服務粒度」是加分）
2. **為什麼「兩邊都有對方沒有的寫入」就不能直接切回去？**（提示：無害合併函數不存在；對照購物車聯集為何在此失效）
3. **如果由你配置 failover 自動化，這場事故給你什麼約束？**（提示：跨區提升需人工確認／區內自動、區間受限；「能力與權限對齊」）
