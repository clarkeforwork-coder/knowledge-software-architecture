# CAP 與 PACELC：被誤讀最多的定理

## 前言

「CAP 就是一致性、可用性、分區容忍三個選兩個嘛，我們選了 AP。」——這句面試標準答案，只要被追問一句「那你們平常不分區的時候，在取捨什麼？」就會當場卡住。CAP 大概是分散式領域被引用最多、也被誤讀最多的定理：多數人記住的「三選二」，恰好是定理**沒有**說的事。

第三章結尾說過，Bounded Context 之間靠事件同步、最終一致——一旦系統跨出單機，這組取捨就跟著你了。這篇筆記把定理還原成它真正說的話，再用 PACELC 補上它沒說的那一半：你每天都在做、卻常常沒意識到的取捨。

## 技術背景

### 先破除誤解：P 不是選項，是天氣

CAP 由 Eric Brewer 於 2000 年提出猜想、Gilbert 與 Lynch 於 2002 年正式證明。三個字母的精確含義：

- **C（Consistency）**：這裡指**線性一致**（linearizability）——所有節點看起來像同一份資料，寫入完成後任何讀取都看得到。比日常口語的「一致性」嚴格得多。
- **A（Availability）**：每個**沒有故障**的節點都必須在有限時間內回應——不能說「等分區好了再說」。
- **P（Partition tolerance）**：節點之間的訊息可能丟失或延遲——也就是網路會斷。

關鍵在 P：**網路分區不是你能拒選的選項，是必然發生的天氣**。交換機重啟、跨機房光纖被挖斷、GC 停頓長到被當成失聯——只要系統跨越一條網路，P 就已經替你勾了。所以定理的真正形狀不是三選二，而是：

> **當分區發生時**，你只能在 C 和 A 之間選一個：拒絕服務以保住一致（CP），或繼續服務並接受分歧（AP）。

這也是為什麼「我們是 CA 系統」是一句沒有意義的話——除非你的「分散式系統」只有一台機器。

### CAP 沒說的那一半：PACELC

CAP 只描述了分區那幾分鐘的事。但系統 99.9% 的時間沒有分區——那時候就沒有取捨了嗎？Daniel Abadi 在 2012 年提出 PACELC 補上這一半：

> if **P**artition → **A** vs **C**；**E**lse → **L**atency vs **C**onsistency

沒有分區的日常，取捨換了一組面孔：**延遲 vs 一致性**。寫入要不要等所有複本確認？等（EC）——每次寫入都付跨節點往返的延遲；不等（EL)——快，但讀到舊資料成為日常可能。這個取捨**每一次寫入都在發生**，跟分區無關。主從複寫要不要開同步模式、讀流量要不要分給 replica，全是 PACELC 的 E 側問題。

### 粒度：不是整個系統選一次

Brewer 本人在〈CAP Twelve Years Later〉（2012）強調的修正：C/A 的選擇**不是系統級的一次性站隊，而是逐資料、逐操作的**。同一個系統裡，購物車可以 PA/EL、扣款可以 PC/EC。「我們是 AP 系統」的第二層錯誤就在這：它假設了一個不存在的統一粒度。

| | 分區時（P） | 平時（E） | 典型例子 |
|---|---|---|---|
| PA/EL | 保可用 | 保延遲 | 購物車、瀏覽紀錄、DNS |
| PC/EC | 保一致 | 保一致 | 扣款、庫存、發號器 |
| PA/EC | 保可用 | 保一致 | 平時等複本、分區時降級續服務 |
| PC/EL | 保一致 | 保延遲 | 平時走快路徑、分區時寧停不錯 |

## 實際案例

同一個電商系統裡的兩類資料，推演分區發生的那十分鐘：

**購物車——選 PA/EL**。機房間斷線，兩邊都繼續接受「加入購物車」。恢復後同一使用者的購物車在兩側有分歧：A 側加了雨傘、B 側加了襪子。合併策略用**聯集**：寧可多出一件讓使用者自己刪，不可弄丟他想買的東西。這正是 Dynamo 系（Amazon 購物車的原型場景）的經典選擇——分區那十分鐘照常做生意，代價是一點點事後和解的尷尬。

**帳戶餘額——選 PC/EC**。同一場分區，若兩側都繼續扣款，恢復後會發現同一筆餘額被花了兩次——這種分歧**沒有無害的合併函數**，「聯集」在錢上叫超扣。所以分區時少數側直接拒絕扣款服務：使用者看到「交易暫時無法使用」，客服接到抱怨，但帳是對的。可用性的損失換正確性，這筆交換在會議室裡就要講好，而不是留給恢復後的對帳程式。

兩類資料、同一場分區、兩種相反的正確答案——**「我們選 AP」這句話在這個系統裡沒有指涉對象**。而平時呢？購物車寫入不等跨機房複本（EL），餘額變動同步寫兩份才回應（EC）：分區十年遇不到幾次，E 側的取捨每秒都在付錢。

## 技術優缺點

把 CAP／PACELC 當**決策語言**（而不是背誦題）使用：

**優勢**

- **逼問粒度**：「這**類**資料分區時怎麼辦」比「我們系統選什麼」是好得多的問題。
- **照亮日常取捨**：E 側把「要不要等複本」從 DBA 的預設參數升級為顯性的架構決策——它本該對照〈[品質屬性場景](../01-architecture-thinking/quality-attributes-and-tradeoffs.md)〉被量化討論。
- **提供止爭的精確詞彙**：「最終一致」「線性一致」有定義可查，比「差不多一致」可仲裁。

**代價**

- **術語被濫用的雙面刃**：CP／AP 標籤貼在真實資料庫上常常兩邊都不精確（Kleppmann 對此有著名的批評——真實系統的故障模式比兩個字母豐富得多），標籤思維會給人虛假的安全感。
- **模型的極端化**：CAP 的 C 是最嚴格的線性一致、A 是每節點必答，真實系統多半活在兩者之間的灰階——這正是下一篇一致性光譜要處理的。

**這改變了什麼實務決策**：資料分類表多兩欄——「分區時：續服務／拒服務？」「平時：等複本／不等？」每一類資料填完這兩格，系統的 CAP 姿態才算真的決定過；填不出來的那幾格，就是還沒被想過的事故劇本。

## 小結

- CAP 不是三選二：P 是必然的天氣，定理只說「分區發生時，C 與 A 二選一」；「CA 系統」是無意義的說法。
- CAP 的 C 是線性一致、A 是每個健康節點必答——都比口語定義嚴格，引用前先對表。
- PACELC 補上沒有分區的日常：延遲 vs 一致性，每次寫入都在取捨，這才是你天天付的帳單。
- 選擇的粒度是逐資料、逐操作，不是系統級站隊——購物車與餘額在同一系統裡給出相反答案。
- 定理裡那個 C 被壓縮成了一個字母，但「一致性」其實是一道光譜——強到什麼程度、弱到什麼程度、中間有哪些工程甜蜜點？下一篇：〈[一致性模型：從強一致到最終一致](consistency-models.md)〉。

## 常見面試題

1. **「CAP 三選二」錯在哪裡？**（提示：P 不是選項；定理只描述分區時刻；說得出「CA 無意義」就到位）
2. **PACELC 比 CAP 多說了什麼？**（提示：Else 側——無分區時的延遲 vs 一致性；舉「要不要等複本確認」即可）
3. **為什麼說不該給資料庫貼 CP／AP 標籤？**（提示：粒度（逐資料）＋真實系統的一致性與故障模式是光譜不是二元）
