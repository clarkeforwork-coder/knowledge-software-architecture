# 一致性模型：從強一致到最終一致

## 前言

客服單：「使用者說發文成功後回到列表，自己的文章不見了；再發一次，變成兩篇重複的。」工程師檢查：寫入成功、複寫正常、延遲 200ms——**一切都符合設計**。這就是問題：設計裡寫著「最終一致」，但沒有人問過「使用者自己看不看得到自己剛做的事」。

[上一篇](cap-and-pacelc.md)把一致性壓縮成 CAP 裡的一個字母 C；這一篇把它展開成光譜。重點不是背出所有模型的名字，而是找到工程上的甜蜜點——**多數系統不需要全域強一致，但幾乎所有系統都需要比「最終一致」多一點的東西**。

## 技術背景

### 先破除誤解：「最終一致」幾乎不承諾任何事

最終一致（eventual consistency，Werner Vogels 2008 年的同名文章使其普及）的完整承諾是：**如果停止寫入，所有複本終將收斂到同一個值**。注意兩個洞：它沒說「終將」是多久（200ms 或 2 小時都合規），也沒說收斂前你會讀到什麼（舊值、更舊的值、來回跳動都合規）。把「最終一致」當成一個具體保證來設計，等於在合約上簽了一個空白條款。

所以真正的問題不是「強一致還是最終一致」，而是：**在收斂之前，你的使用者被承諾了什麼？**

### 光譜：從最強到最弱

| 模型 | 承諾 | 直覺說法 |
|---|---|---|
| 線性一致（linearizability） | 全系統像單機：寫入完成後，任何人立刻讀到 | 「只有一份資料」的幻覺 |
| 順序一致（sequential） | 所有人看到**同一套**操作順序，但不保證即時 | 大家看同一部片，但可能慢播 |
| 因果一致（causal） | 有因果關係的操作，人人按因果序看到 | 回覆不會出現在原文之前 |
| Session 保證 | 只對「同一個使用者的會話」給承諾（見下） | 自己至少要看得到自己 |
| 最終一致（eventual） | 停止寫入後終將收斂 | 只承諾「不會永遠錯下去」 |

越往上承諾越強、代價越高：線性一致需要跨節點協調，寫入延遲直接吃 PACELC 的 EC 帳單；越往下越便宜，但應用層要自己收拾分歧。

### 工程甜蜜點：Session 保證

Terry 等人 1994 年提出的四條 session guarantees，是這條光譜上投報率最高的一段——它們不承諾全世界一致，只承諾**單一使用者的視角自洽**：

- **讀己之寫（read your writes）**：我寫完，我自己一定讀得到——開頭那張客服單缺的就是這條。
- **單調讀（monotonic reads）**：我讀到過新值，就不會再讀到更舊的——不會「重新整理後文章倒退嚕」。
- **單調寫（monotonic writes）**：我的兩次寫入按順序生效。
- **讀後寫（writes follow reads）**：我基於某次讀的寫入，排在那次讀之後——回覆不會插到原文前面。

關鍵洞察：**使用者感知到的「不一致」，九成是 session 保證的缺失，不是全域一致的缺失**。使用者不在乎別人晚 300ms 看到他的發文；他在乎自己看不到。而 session 保證比全域強一致便宜一個數量級——不需要共識協定，只需要路由上的一點心思。

### 因果一致：多人互動的底線

跨出單人視角，下一個常見需求是因果：留言板上「問題」與「回答」的順序、聊天室裡「撤回」與「原訊息」的順序。因果一致只對**有依賴關係**的操作排序，無關的操作各走各的——比全域排序便宜，又足以避免「先看到回答、後看到問題」的荒謬。

## 實際案例

讀寫分離（一主多從）的內容平台，複寫延遲平均 200ms，尖峰 3 秒。開頭那張客服單的完整病理與三種處方：

**病理**：發文寫主庫 → 302 轉向列表頁 → 列表查詢被負載均衡導到 replica → 複寫還沒到 → 自己的文章不在列表上 → 使用者判定「發失敗」→ 重發 → 兩篇。第二篇成功寫入也複寫成功——**系統的每個元件都正常，壞掉的是承諾的缺位**。

**處方一：寫後讀主**。該使用者寫入後 N 秒內的讀取都路由到主庫（N 蓋過複寫延遲的高分位）。實作簡單（cookie 記一個 `wrote_at` 時間戳）；代價是寫入活躍時段主庫讀壓上升，以及 N 的調校——設短了破功，設長了主庫變回單點熱區。

**處方二：session 黏著**。同一 session 固定讀同一台 replica，並確保該 replica 至少追上這個 session 自己的寫入位點（帶著複寫位置 token 走）。承諾更精確（正是「讀己之寫＋單調讀」的教科書實作）；代價是路由層要理解複寫位點，架構複雜度上一階。

**處方三：寫入回填**。發文成功後，前端直接把剛寫的內容插進列表（不等後端查詢）。零後端改動、體感最快；代價是這只是**視覺上的**讀己之寫——換個裝置就破功，且前端與後端的資料建構邏輯出現複本。

三種處方沒有一種是「升級到強一致」——把資料庫改成同步複寫（EC）當然也能修，但那是用**每一次寫入的延遲**去付**只有寫入者本人在乎的承諾**，帳算不平。對症的藥是 session 保證，這正是光譜存在的意義：**病灶在哪一檔，藥就開到哪一檔**。

## 技術優缺點

按光譜選檔（而不是全系統一刀切）：

**優勢**

- **成本對症**：只在需要的路徑上付一致性的錢——錢包強一致、動態牆最終一致、個人視角 session 保證。
- **承諾可言說**：「這條路徑保證讀己之寫」是可以寫進設計文件、可以驗收的句子；「應該很快就一致了」不是。
- **與業務對話有介面**：光譜給了非工程方選項——「晚 5 分鐘可以嗎？自己看不到自己可以嗎？」對照〈[品質屬性](../01-architecture-thinking/quality-attributes-and-tradeoffs.md)〉的追問法。

**代價**

- **心智負荷**：每條資料路徑掛著不同承諾，新人需要一張地圖；文件不更新時，地圖比沒有更危險。
- **測試困難**：本機與測試環境的複寫延遲趨近於零，一致性 bug 只在生產尖峰現形——需要故障注入或延遲模擬才能在上線前看到。
- **路由複雜度**：session 保證的實作把「一致性」的重量從資料庫挪到了路由層，那裡從此不再是無腦轉發。

**這改變了什麼實務決策**：評審資料路徑時的問題從「這裡要不要強一致」（二元、昂貴、常被拒）變成「**這裡的讀者是誰？他剛剛寫過什麼？他看到舊值會做出什麼錯誤行為？**」——答案自然落在光譜的某一檔，而那一檔幾乎總是比「全域強一致」便宜、比「純最終一致」負責。

## 小結

- 「最終一致」只承諾停止寫入後終將收斂——沒說多久、沒說收斂前讀到什麼；把它當具體保證是空白合約。
- 一致性是光譜：線性 → 順序 → 因果 → session 保證 → 最終；越強越貴，貴在延遲與協調。
- 工程甜蜜點是 session 保證：使用者感知的不一致九成是「自己看不到自己」，而這比全域強一致便宜一個數量級。
- 讀寫分離系統的標配自問：讀己之寫由誰保證？答不出來，客服單已經在路上。
- 對症下藥：病灶在哪一檔，藥開到哪一檔——用同步複寫去修「自己看不到自己」是帳算不平的豪購。
- 最終一致的世界還有一個伴生現實：訊息會重來、請求會重送——同一件事被執行兩次時，系統還對嗎？下一篇：〈[冪等性：重試安全的前提](idempotency.md)〉。

## 常見面試題

1. **「最終一致」到底承諾了什麼、沒承諾什麼？**（提示：停止寫入後終將收斂；沒有時限、沒有收斂前的讀取保證）
2. **讀寫分離下怎麼實作「讀自己的寫」？**（提示：寫後讀主、session 黏著＋複寫位點、前端回填——說得出各自代價更好）
3. **哪些資料你會堅持線性一致？**（提示：無法無害合併分歧的——錢、庫存、唯一性發號；並說明其餘為何降檔）
