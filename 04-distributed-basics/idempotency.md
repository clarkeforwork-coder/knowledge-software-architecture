# 冪等性：重試安全的前提

## 前言

客訴：「我被扣了兩次款。」查了半天，付款服務沒有 bug：第一次請求成功了，只是回應在半路逾時；APP 判定失敗、自動重送；第二次也成功了。**每個元件都正確，錢就是扣了兩次。**

這是分散式系統最樸素也最兇的一課：**逾時不等於失敗**。一次呼叫的結局有三種——成功、失敗、**不知道**——而網路只要夠長，「不知道」就是日常。消解「不知道」的唯一辦法是再試一次；而讓「再試一次」不變成「再扣一次」的唯一辦法，就是這篇的主題：冪等性（idempotency）。

## 技術背景

### 先破除誤解：「我沒寫重試邏輯，所以不需要冪等」

重試無所不在，而且**大多不是你寫的**：使用者不耐煩重按按鈕、瀏覽器與 APP 的自動重送、API 閘道器的 retry 策略、訊息佇列的 at-least-once 重投遞（〈[事件驅動架構](../02-architecture-styles/event-driven-architecture.md)〉的必答題之一）、批次任務失敗重跑、維運人員的「再執行一次看看」。你的寫入介面暴露在世界上的那一刻，就已經活在重試的雨裡了——問題從來不是「會不會收到重複請求」，是「收到時對不對」。

更根本的：訊息傳遞層只有兩種誠實的保證——至多一次（at-most-once，可能丟）與至少一次（at-least-once，可能重）。**傳輸層的 exactly-once 不存在**；工程上的「恰好一次」是合成品：**at-least-once ＋ 接收端冪等 ＝ 效果上恰好一次（effectively once）**。

### 定義與判別

冪等：同一個操作執行一次和執行 N 次，對系統狀態的影響相同——`f(f(x)) = f(x)`。

| 操作 | 冪等？ | 原因 |
|---|---|---|
| `UPDATE SET status = 'PAID'` | ✅ | 設絕對值，重放不變 |
| `DELETE FROM ... WHERE id = 1` | ✅ | 刪掉的東西再刪還是不在 |
| `UPDATE SET balance = balance - 100` | ❌ | 相對運算，重放一次扣一次 |
| `INSERT INTO orders (...)` | ❌ | 重放多一筆 |
| HTTP GET／PUT／DELETE | ✅（規範上） | HTTP 語意即如此設計 |
| HTTP POST | ❌ | 這正是「下單」們住的地方 |

麻煩之處一目了然：**業務上最要緊的操作（扣款、下單、發貨），天生都是非冪等的**。所以冪等不是挑出天生安全的操作，而是替天生危險的操作**加裝**安全機制。

### 工具箱：四種實作手段

1. **冪等鍵（idempotency key）**——通用解。呼叫端為每個「業務意圖」產生唯一鍵（重試沿用**同一把**），服務端以唯一約束記錄之：第一次執行並存下結果；重複到達時不再執行，**回放第一次的結果**。Stripe 的 `Idempotency-Key` header 是公開範本。
2. **狀態機守衛**——用生命週期擋重放：只有 `PENDING → PAID` 是合法遷移，第二次扣款請求到達時訂單已是 `PAID`，遷移不成立、直接拒絕。適合有明確狀態流的聚合（〈[戰術設計](../03-ddd/tactical-design-aggregate.md)〉的 `ensureModifiable` 同款思路）。
3. **條件更新**——把「期望的前提」寫進更新條件：`UPDATE ... WHERE version = 42`（樂觀鎖）或 `WHERE status = 'PENDING'`，影響列數為 0 即知重放或衝突。單機交易與鎖的機制細節見 knowledge-java 的〈[交易與資料庫鎖](../../knowledge-java/08-data-access/transactions-and-database-locks.md)〉。
4. **改寫成絕對語意**——能把「扣 100」改成「設為 900」的地方就改；把相對運算移到讀取端計算。適用面窄，但成本最低。

### 兩個常被漏掉的設計題

- **重複請求要「回放」，不是「報錯」**。呼叫端收到「重複請求」錯誤時無從行動——它重試正是因為不知道第一次的結局。正確行為是把第一次的回應原樣再給一次，讓呼叫端無法區分（也不需要區分）自己是第幾次成功。
- **鍵的範圍與壽命**。鍵綁定的是「一次業務意圖」不是「一次 HTTP 請求」（使用者改了金額再送，是新意圖、新鍵）；存留期要蓋過最長的重試窗口，並有清理策略——冪等表是會長大的。

## 實際案例

扣款 API 的時間線推演：

```
APP            付款服務           資料庫
 │─ 扣款 $500 ──>│                  │
 │               │─ 執行成功 ───────>│   ← 錢已扣
 │      ✗ <──────│  （回應在網路上死掉）
 │ 逾時！
 │─ 重送 $500 ──>│                  │
 │               │─ 再執行一次 ─────>│   ← ❌ 雙扣，客訴在路上
```

加上冪等鍵之後，服務端的骨架：

```java
// APP 端：同一次付款意圖，重試永遠帶同一把鍵
POST /payments
Idempotency-Key: 7f3d...a1   // 進入付款頁時產生，重送不換

// 服務端：先佔位，再做事，結果落檔
try {
    idempotencyRepo.insert(key, IN_PROGRESS);      // 唯一約束佔位
} catch (DuplicateKeyException e) {
    var rec = idempotencyRepo.find(key);
    if (rec.isCompleted()) return rec.response();  // ✅ 回放第一次的結果
    return Response.status(409).retryAfter(2);     // first 還在跑：稍後再來，別並行執行
}
charge(...);                                       // 真正扣款
idempotencyRepo.complete(key, response);           // 結果與鍵同庫同交易落檔
return response;
```

三個容易踩的細節都在骨架裡了：**先佔位再執行**（否則兩個並發重試會同時通過檢查——檢查與佔位必須是同一個原子動作，靠唯一約束而不是先查後插）；**進行中要擋並行**而不是回錯誤結果；**結果與冪等紀錄同交易寫入**（否則「扣款成功、結果沒存下」的半途狀態會讓回放無物可放）。時間線重播：第二次請求撞上唯一約束，拿回第一次的成功回應——APP 只知道「付款成功一次」，世界如常。

## 技術優缺點

**優勢**

- **重試從賭博變成安全操作**：逾時的「不知道」可以放心用重試消解，這是一切重試機制、事件驅動、Saga 補償的共同地基。
- **呼叫端被簡化**：不需要「先查上次成功了沒」的防禦舞蹈，帶鍵重送就是完整協議。
- **與 at-least-once 合成 effectively once**：用應用層的一張表，補上傳輸層做不到的承諾。

**代價**

- **鍵的治理成本**：誰產生、怎麼傳遞、綁什麼範圍、活多久——每一題答錯，防護就有洞（最常見：重試時換了新鍵，等於裸奔）。
- **儲存與清理**：冪等表隨流量成長，回放要求存下完整回應，過期清理是必須的維運項。
- **鏈路上每一跳各自負責**：A 冪等不代表 A 呼叫的 B 安全——每個寫入服務都要自己那份防護，沒有全鏈路的免費傳染。

**這改變了什麼實務決策**：冪等鍵從「實作細節」升格為 **API contract 的一部分**——對外的寫入介面在設計評審時就要回答「重複請求的行為是什麼」，如同回答認證與限流；訊息消費者的驗收清單裡，「重投遞測試」與功能測試同級。

## 小結

- 呼叫的結局有三態：成功、失敗、不知道；「不知道」只能靠重試消解，重試的安全靠冪等。
- 重試大多不是你寫的：使用者、閘道器、佇列、批次都在替你重試——寫入介面天生活在重試的雨裡。
- 傳輸層沒有 exactly-once：at-least-once ＋ 冪等 ＝ effectively once，這是工程上唯一的合成路徑。
- 工具箱四件：冪等鍵＋唯一約束（通用）、狀態機守衛、條件更新、絕對語意；先佔位再執行、重複要回放不報錯。
- 冪等鍵是 API contract 的一部分，不是實作細節。
- 單一操作安全了；但「下單 → 扣庫存 → 扣款」是一**串**跨服務的寫入，中間斷掉怎麼辦？本章壓軸：〈[分散式交易：2PC、Saga 與 Outbox](distributed-transactions.md)〉。

## 常見面試題

1. **為什麼說訊息系統的 exactly-once 不存在？那大家宣稱的是什麼？**（提示：兩將軍式的確認困境；at-least-once＋冪等消費＝effectively once）
2. **冪等鍵怎麼設計？綁請求還是綁業務意圖？**（提示：意圖；重試同鍵、新意圖新鍵；服務端唯一約束＋回放）
3. **併發的兩個相同請求同時通過「查無此鍵」的檢查，怎麼防？**（提示：檢查與佔位要原子——唯一約束先佔位，不是先查後插）
